# 正确性、契约与语义回归检查

本参考文档用于帮助判断：一次改动是否不仅改变了实现方式，还改变了原有行为语义、完成条件、调用契约、默认值语义、兼容性边界或业务规则一致性。

## 1. 适用范围

本参考适用于以下类型的改动：

- 函数 / 方法签名、参数顺序、默认值、返回结构、异常语义变化
- DTO / schema / 配置项 / 状态码映射 / 枚举语义变化
- 调用入口切换、统一入口被绕过、分层边界变化
- 同步 / 异步切换、完成条件变化、状态机迁移变化
- 业务规则在多入口、多模块、多分支中的一致性问题
- 重大逻辑变更但验证信号不足的场景

当审查重点落在“行为是不是变了”“契约有没有破”“调用方还能不能成立”时，应优先参考本文件。

## 2. 核心原则

审查时不要停留在“实现方式变了没有”，还要追问：

- 返回时机是否变化
- 完成条件是否变化
- 默认值语义是否变化
- 状态迁移语义是否变化
- 调用入口是否从受控路径切换为非受控路径
- 异常 / 失败后的可观察行为是否变化
- 上下游调用方是否仍然满足原有预期
- 业务规则、配置语义、阈值映射是否仍在多处保持一致

不要把“实现不同”“写法变了”“结果可能略有差异”直接等同于语义回归。只有当**旧语义、现语义、影响路径**能够相对直接地说明出来时，才适合升级为正式问题。

## 3. 高优先级问题（优先报告）

### 接口契约或返回语义被破坏
- **何时报告**：参数顺序、默认值、返回结构、异常语义、schema 字段、配置项含义变化后，调用点、上下游或兼容层没有同步更新时。
- **为什么重要**：这类问题容易造成直接回归，且往往影响范围比单文件更大。
- **修复建议**：恢复原契约，或同步更新所有调用点、兼容层与测试。

### 统一入口 / 仓库层 / 服务层契约被绕过
- **何时报告**：原先通过统一入口、控制层、服务层、仓库层完成的动作，被改到更外层直接执行，导致验证、审计、状态约束或业务规则被绕过时。
- **为什么重要**：问题不只是“写法变了”，而是原有语义和边界可能被打破。
- **修复建议**：恢复统一入口，或在新入口显式补齐原有契约、校验与测试。

### 默认值、阈值、状态映射或容错语义变化未联动
- **何时报告**：默认值、阈值、状态码映射、错误兜底、容错分支发生变化，但未见调用方、配置说明、上下游或测试同步更新时。
- **为什么重要**：这类问题常导致静默回归，不容易第一时间暴露。
- **修复建议**：恢复原语义，或同步更新配置说明、调用方与回归测试。

### 重大逻辑变更缺少必要验证信号
- **何时报告**：状态机、结算、权限、数据转换、批处理等核心路径发生较大改动，但缺少必要测试、断言、联动更新或回归验证信号时。
- **为什么重要**：高风险路径的大改动在没有验证支撑时，回归成本很高。
- **修复建议**：补齐关键回归场景测试、断言和联动验证说明。

## 4. 中优先级问题（按上下文报告）

### 业务规则在多入口 / 多模块中不一致
- **何时报告**：相同业务规则在不同入口、分支或模块中的实现已出现明显偏差，且当前改动直接引入、放大或暴露了这种不一致时。
- **为什么重要**：容易导致某些路径正确、某些路径错误，后续修改也容易修一处漏一处。
- **修复建议**：收敛到统一判断入口或共享规则实现。

### 同步改异步、完成语义变化但调用方未承接
- **何时报告**：同步流程改为异步流程，但等待条件、回调时机、提交时机或上层调用约定没有同步更新时。
- **为什么重要**：这类问题未必立即抛异常，但常引发时序回归。
- **修复建议**：恢复原完成语义，或同步修改调用契约与测试。

## 5. 风险提示而非正式问题的场景

以下情况更适合报为风险提示：

- 是否回归仍取决于运行环境、外部配置或隐藏上下文
- 能看出语义可能变化，但缺少足够调用链证据
- 只能说明实现路径变了，但还不能稳定推出可观察行为已经改变

## 6. 输出建议

描述这类问题时，优先写清：
1. 以前语义是什么
2. 现在语义变成了什么
3. 为什么这种变化会影响调用方、状态一致性或正确性
4. 建议如何恢复契约或同步更新上下游

若多个后果都来自同一次语义变化，默认合并成一条主问题，在影响分析中展开，不要把同一语义回归拆成多条正式问题。

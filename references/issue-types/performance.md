# 性能检查

## 1. 适用范围

本参考适用于 `code-reviewer-optimized` 在服务端、前端、脚本、数据处理和基础设施代码中的性能审查，重点覆盖：

- 热点请求路径、批处理任务、列表页、搜索页、报表、导入导出
- 算法复杂度、数据库访问、缓存、渲染、网络调用与序列化开销
- 重复计算、无界查询、N+1、全量加载、阻塞调用、资源放大
- 与响应变慢、吞吐下降、资源耗尽、页面卡顿和成本放大相关的风险点

当变更涉及循环处理、大数据量列表、数据库查询、缓存、远程调用、渲染热点或后台任务时，应优先参考本文件。

## 2. 审核目标

做性能审查时，优先关注：

- 是否存在已经足以影响当前路径的性能问题，而不是泛泛地追求“更快”
- 是否有明确的触发条件、数据规模、调用频率或热点路径支撑该问题
- 是否存在明显会放大成本的查询、渲染、序列化或并发模型错误
- 是否能给出低歧义、可执行、与收益匹配的修复建议
- 是否避免把理论优化点当成主要审查意见

## 3. 高优先级问题（优先报告）

### 明显的 N+1 查询或重复远程调用
- **何时报告**：列表、批处理、模板渲染、导出任务中，对每个元素重复发起数据库查询、RPC、HTTP 调用，且成本会随数据量线性放大时。
- **为什么重要**：这是最常见、最容易在真实业务中产生明显性能问题的模式之一。
- **建议如何审**：关注循环中的查询 / 请求、ORM 关联懒加载、模板中访问关联字段、逐项补充详情逻辑。
- **修复建议**：改为批量查询、预加载、JOIN、缓存或一次性拉取所需数据。
- **何时给代码示例**：建议给。

### 无分页、无 limit、全量扫描或全量加载
- **何时报告**：面向用户的接口、后台列表、搜索、导出、定时任务默认拉取全表或全量对象，且缺少分页、limit、流式处理时。
- **为什么重要**：会直接放大数据库、内存和响应时间成本。
- **建议如何审**：查看列表接口、管理页、统计接口、搜索、导出、批量任务、文件读写。
- **修复建议**：增加分页、limit、游标、分批处理或流式消费。
- **何时给代码示例**：建议给。

### 热点路径存在明显不必要的高复杂度实现
- **何时报告**：在高频或大数据量路径中，使用 O(n²) 或更差复杂度实现，而存在明显更低复杂度替代方案时。
- **为什么重要**：在真实数据规模下会造成可感知变慢甚至阻塞。
- **建议如何审**：关注双重循环查重、嵌套过滤、循环内线性查找、重复排序和聚合。
- **修复建议**：改用哈希结构、索引映射、预处理、批量聚合或更合适的数据结构。
- **何时给代码示例**：建议给。

### 渲染热点中重复重算或不稳定渲染边界
- **何时报告**：前端热点页面在每次渲染中重复执行昂贵过滤、排序、对象构造，或大列表无虚拟化导致明显卡顿时。
- **为什么重要**：会直接影响交互流畅性和用户体验。
- **建议如何审**：关注列表页、图表页、搜索联想、表单联动、模板 / 组件中的重复计算。
- **修复建议**：缓存派生结果、拆分组件边界、减少重复构造、按需虚拟化或延迟加载。
- **何时给代码示例**：建议给。

### 大对象 / 大文件 / 大结果集处理方式无界
- **何时报告**：一次性把大文件读入内存、全量反序列化大对象、返回超大响应、无上限缓存聚合时。
- **为什么重要**：容易导致内存峰值过高、GC 压力、进程抖动或直接 OOM。
- **建议如何审**：关注导入导出、日志处理、报表生成、附件预览、批量接口、聚合缓存。
- **修复建议**：改为流式处理、分块读取、分页聚合或限制响应规模。
- **何时给代码示例**：建议给。

### 外部依赖调用缺少超时 / 并发边界，导致性能放大为稳定性问题
- **何时报告**：远程调用无超时、无限重试、并发请求无上限或串行等待多个外部依赖时。
- **为什么重要**：这类问题常同时带来性能劣化和系统不稳定。
- **建议如何审**：查看 HTTP / RPC / 存储 / 队列调用、聚合接口、批处理任务、回调链路。
- **修复建议**：增加超时、并发限制、批量化、降级策略或合理并发执行。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### 缓存策略缺失或失效策略明显不合理
- **何时报告**：热点读路径重复命中数据库 / 远程服务，或缓存键、TTL、失效策略明显会造成高成本或脏读风险时。
- **为什么重要**：缓存问题常有收益，但必须结合访问模式和一致性要求判断。
- **修复建议**：根据读写比例和一致性要求补充缓存，并明确失效策略。

### 序列化、反序列化或格式转换开销过高
- **何时报告**：同一数据在单条链路中重复 JSON 编解码、对象复制、字段转换，已构成明显浪费时。
- **为什么重要**：会增加 CPU 开销和响应延迟。
- **修复建议**：减少重复转换，复用结果，避免不必要的中间表示。

### 容器、集合或内存分配策略不合适
- **何时报告**：热点路径中存在明显的大量重复分配、频繁扩容、大对象复制，但暂未到必须阻断合并的程度时。
- **为什么重要**：会逐步放大成本，但要结合路径和规模判断。
- **修复建议**：预分配、复用缓冲区、改用更合适的数据结构。

### 非热点路径的结构复杂度或重复对象构造已开始影响修改安全性
- **何时报告**：虽然不在明确热点路径上，但函数过长、嵌套过深、重复分支、一次性对象反复构造等问题已经影响理解、排障或后续安全修改时。
- **为什么重要**：这类问题不一定立刻表现为性能瓶颈，但会放大后续改动的误改风险，也会让真正的性能问题更难识别。
- **修复建议**：拆分职责、收敛重复分支、提取共享构造逻辑，仅在收益明确时再做局部性能优化。

### 前端资源加载策略欠佳
- **何时报告**：首屏加载了明显不必要的大模块、大图片或大列表渲染，且会实质影响页面打开速度时。
- **为什么重要**：会拖慢真实用户体验，但要避免机械套“代码分割模板”。
- **修复建议**：按需加载、懒加载、图片优化或拆分关键渲染路径。

## 5. 低优先级问题（谨慎报告）

### 一般性微优化建议
- **何时报告**：仅当它出现在已知热点路径，或项目明确在意该类成本时。
- **为什么重要**：像减少一次小对象创建、替换某个语法糖之类，默认不应成为主要审查意见。

### 理论上的更优算法或更优框架特性
- **何时报告**：仅当当前实现已经在目标场景下明显不够用时。
- **为什么重要**：不要把“存在更快写法”当成“当前实现有问题”。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非已造成明确问题：

- 看到循环就泛泛地说“复杂度太高”
- 看到对象创建就泛泛地说“GC 压力大”
- 看到没有缓存就默认认为应该加缓存
- 看到前端列表就机械要求上虚拟列表
- 看到导入整个库就机械说 bundle 太大，但没有说明实际影响路径
- 没有数据规模、调用频率、热点路径支撑的性能臆测
- 与当前改动无关的历史性能债

## 7. 性能审查特有的判断重点

### 先确认它是不是热点
- 这个逻辑是否在高频请求、批处理、大列表、导出、搜索、渲染主路径上
- 数据规模是否足以让问题显现
- 调用频率是否足以放大成本

### 性能问题要尽量说明“为什么现在值得提”
- 是用户可感知卡顿
- 是数据库 / CPU / 内存成本明显放大
- 是会随着数据规模增长快速恶化
- 是已经接近稳定性问题，而不是纯理论优化点

### 区分阻断项和优化项
- 阻断项通常是 N+1、全量加载、热点 O(n²)、大对象无界处理、远程调用失控
- 优化项通常是轻微重复计算、局部资源加载、容器选择改进等

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- N+1 查询
- 热点路径高复杂度实现
- 大结果集 / 大文件无界处理
- 渲染热点重复重算
- 外部调用超时 / 并发模型明显错误
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 一般性缓存建议
- 轻度资源加载优化
- 容器和数据结构微调
- 非关键路径的小型性能建议

## 9. 审核输出建议

- 只报告那些在当前改动里“值得现在就处理”的性能问题
- 提性能问题时尽量说明触发路径、数据规模、调用频率或成本放大机制
- 如果没有证据支撑，不要把性能猜测写成确定结论
- 不要把性能优化手册逐条翻译成 review 评论
- 如果没有发现值得报告的高置信度性能问题，应明确说明

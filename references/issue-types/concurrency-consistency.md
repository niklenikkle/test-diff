# 并发与一致性检查

本参考文档用于帮助判断：并发问题是否已经由多项证据组合成立，而不是只因为出现了某个异步 API、线程池、协程或 fan-out 调用就直接下结论。

## 1. 适用范围

本参考适用于以下类型的改动：

- 并发执行路径、异步任务、线程池、协程、后台任务
- 共享状态、共享容器、缓存、懒加载、初始化 / 销毁流程
- fan-out / fan-in 并发汇总、并行 RPC / HTTP / IO 调用
- 顺序依赖、检查-执行间隙、原子性与状态一致性要求

当改动涉及“多个执行单元是否会交错”“共享对象是否会被并发访问”“状态是否还能保持一致”时，应优先参考本文件。

## 2. 核心原则

并发问题通常需要组合判断以下几个维度：

- 是否存在并发执行路径
- 是否存在共享可变状态
- 共享对象 / 容器本身是否线程安全
- 是否存在锁、原子操作、统一汇总发布或等效同步保护
- 是否存在顺序依赖、检查-执行间隙或状态覆盖风险

并发问题的正式成项应尽量围绕**根因**组织：如果多个后果都来自同一并发缺口，应优先合并为一条主问题，在影响分析中展开一致性、崩溃、覆盖或丢失更新风险，而不是拆成多条相似问题。

## 3. 高优先级问题（优先报告）

### 共享可变状态在线程 / 协程并发路径中无保护
- **何时报告**：多个执行单元并发读写同一共享对象、容器、缓存、聚合结果，且未见锁、原子保护或单线程收敛时。
- **为什么重要**：容易直接导致数据竞争、状态覆盖、一致性错误或崩溃。
- **修复建议**：收敛写入点，增加锁 / 原子保护，或改为不可变快照与统一发布。

### 非线程安全对象被并发修改
- **何时报告**：非线程安全容器、客户端、连接、上下文对象在并发路径中被复用或修改时。
- **为什么重要**：这类问题通常有明确后果，且修复方向清晰。
- **修复建议**：避免跨线程共享不安全对象，必要时每个任务独立实例化或在外层串行化访问。

### 顺序依赖、原子性或检查-执行间隙被打散
- **何时报告**：先检查后执行、库存扣减、状态切换、唯一性判断、懒加载初始化等流程在并发下缺少一致性保护时。
- **为什么重要**：会导致重复执行、状态错乱、数据不一致。
- **修复建议**：把关键区段收敛到原子操作、事务或受控串行路径中。

### 并发 fan-out 缺少等待、失败收敛、限流、超时或取消机制
- **何时报告**：并发请求批量打出，但没有统一等待、失败聚合、超时控制、限流或取消逻辑时。
- **为什么重要**：容易把性能问题放大为稳定性和一致性问题。
- **修复建议**：补齐等待 / 收敛机制、并发上限、超时和取消控制。

## 4. 风险提示而非正式问题的场景

以下情况更适合报为风险提示：

- 只能看到异步调用，但无法确认共享状态是否跨执行单元共享
- 无法确认运行模型、调度模型或实际竞争条件
- 只能推测可能有竞态，但没有足够证据说明共享对象与交错操作

## 5. 输出建议

描述这类问题时，优先写清：
1. 并发从哪里产生
2. 共享对象是什么
3. 哪些操作可能并发交错
4. 缺少什么保护
5. 可能导致什么一致性或线程安全后果

不要把同一个并发根因机械拆成“竞态条件”“数据不一致”“顺序错乱”“崩溃风险”多条独立正式问题，除非它们确实来自不同根因或需要不同修复动作。

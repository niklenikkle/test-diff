# 安全检查

## 1. 适用范围

本参考适用于 `code-reviewer-optimized` 在各类语言、框架和服务端 / 客户端代码中的安全审查，重点覆盖：

- 接口、路由、服务层、模板渲染、文件处理、外部请求与后台任务
- 输入边界、认证授权、对象级访问控制、敏感信息处理
- 数据库查询、命令执行、路径处理、反序列化、动态执行
- 与 XSS、注入、越权、SSRF、敏感信息泄露、文件上传相关的风险点

当变更涉及用户输入、资源访问、HTML 输出、数据库访问、文件系统、令牌 / 会话、第三方请求或管理操作时，应优先参考本文件。

## 2. 审核目标

做安全审查时，优先关注：

- 是否存在可被实际利用的高价值漏洞，而不是泛泛而谈“安全最佳实践”
- 是否出现信任边界混乱，导致不可信输入直接进入关键逻辑
- 是否存在认证、授权、对象访问控制不完整的问题
- 是否有敏感信息泄露、危险执行、外部资源访问缺乏约束等明显风险
- 是否能给出明确、可修复、低歧义的建议

## 3. 高优先级问题（优先报告）

### 对象级授权缺失或访问控制失效
- **何时报告**：用户可以通过 ID、slug、路径参数、租户标识等访问或修改不属于自己的资源，或只有“已登录”检查而没有资源级授权时。
- **为什么重要**：这是高价值、常见且影响直接的安全问题。
- **建议如何审**：关注订单、文档、用户资料、组织资源、后台管理接口、导出 / 删除 / 编辑操作。
- **修复建议**：在资源读取和修改入口增加显式授权检查，不要只依赖前端隐藏、路由守卫或页面权限。
- **何时给代码示例**：建议给。

### SQL 注入、命令注入、模板注入或动态执行风险
- **何时报告**：用户输入通过字符串拼接进入 SQL、shell 命令、模板表达式、脚本执行、动态查询条件时。
- **为什么重要**：这类问题一旦成立，危害通常很高，且修复方式明确。
- **建议如何审**：查看原生 SQL、动态筛选 / 排序、`system` / `exec`、脚本拼接、模板求值、规则引擎入口。
- **修复建议**：使用参数化查询、白名单映射、参数隔离，避免把不可信输入直接交给解释器或 shell。
- **何时给代码示例**：安全问题，建议给。

### XSS 或不可信 HTML 渲染
- **何时报告**：用户输入、CMS 内容、Markdown、富文本结果未经净化直接进入 HTML、模板或 DOM 时。
- **为什么重要**：会导致前端脚本执行、会话劫持和数据窃取。
- **建议如何审**：关注 `innerHTML` / `dangerouslySetInnerHTML` / `v-html`、模板原样输出、评论区、公告、富文本渲染。
- **修复建议**：优先按普通文本渲染；确需渲染 HTML 时，使用可信净化流程并限制允许标签。
- **何时给代码示例**：建议给。

### SSRF 或服务端外部请求目标未受控
- **何时报告**：服务端根据用户输入发起 URL 请求，且没有域名白名单、协议限制、内网地址限制或重定向约束时。
- **为什么重要**：可用于探测内网、访问云元数据、绕过边界访问内部服务。
- **建议如何审**：查看 webhook、抓取器、文件预览、URL 代理、图片 / 文档拉取、回调测试功能。
- **修复建议**：增加白名单、协议限制、IP / 网段限制和重定向控制。
- **何时给代码示例**：建议给。

### 路径遍历、文件上传或文件处理边界失守
- **何时报告**：文件路径、压缩包内容、上传文件名、下载目标、导入导出路径直接由用户输入决定，且缺少校验时。
- **为什么重要**：会导致任意文件读取 / 覆盖、恶意文件落地或执行链风险。
- **建议如何审**：关注上传、解压、模板导入、日志下载、附件预览、报表导出。
- **修复建议**：固定基准目录，校验文件类型 / 大小 / 扩展名 / MIME，使用安全文件名，不信任原始路径。
- **何时给代码示例**：建议给。

### 认证 / 会话 / 令牌处理存在明显安全缺口
- **何时报告**：令牌硬编码、会话未失效、敏感操作未重新校验身份、密码明文存储或重置流程可被绕过时。
- **为什么重要**：会直接影响账户安全和整体信任边界。
- **建议如何审**：查看登录、密码重置、token 刷新、登出、会话失效、Cookie 配置、管理操作二次确认。
- **修复建议**：收紧令牌生命周期、确保密码安全存储、补齐关键操作校验和会话失效逻辑。
- **何时给代码示例**：建议给。

### 敏感信息泄露
- **何时报告**：源码、日志、错误响应、前端 bundle、配置文件中出现凭证、token、密码、私钥、内部地址、完整堆栈等敏感内容时。
- **为什么重要**：泄露往往可直接被利用，并放大其他漏洞危害。
- **建议如何审**：查看配置、异常处理、中间件、调试输出、构建产物、客户端环境变量暴露。
- **修复建议**：改用安全配置来源，日志脱敏，生产环境隐藏内部错误细节。
- **何时给代码示例**：建议给。

### 输入边界校验缺失或边界分支过于乐观
- **何时报告**：请求参数、Header、Cookie、消息体、路径参数、外部响应、配置值等不可信数据未经必要校验就进入权限判断、资源访问、路径拼接、查询条件、安全决策或状态变更逻辑时；或空值、缺字段、非法枚举、异常格式被默认值静默兜底时。
- **为什么重要**：很多安全问题不是出在“危险 API”本身，而是边界过于乐观，导致不可信输入在关键路径中获得了超出预期的影响力。
- **建议如何审**：关注控制器/路由入口、DTO/schema、反序列化结果、Webhook/回调、配置加载、第三方响应消费逻辑。
- **修复建议**：在系统边界增加结构校验、类型约束、白名单和语义校验；区分“缺省可接受”和“输入非法”两类情况，不要统一吞成默认值。
- **何时给代码示例**：建议给。

### 不安全反序列化或对不可信数据进行危险解析
- **何时报告**：使用 `pickle`、`unserialize`、不安全 YAML loader、动态对象恢复或脚本化规则解析处理不可信输入时。
- **为什么重要**：可能导致对象注入、任意代码执行或状态污染。
- **建议如何审**：关注缓存恢复、任务队列、配置导入、插件机制、历史兼容格式读取。
- **修复建议**：改用安全格式和受限解析器，并在边界上增加结构校验。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### 安全日志和告警能力不足
- **何时报告**：登录失败、权限拒绝、关键配置变更、异常请求等安全事件没有可用日志，且会明显影响追踪和审计时。
- **为什么重要**：影响检测和追责，但通常不应压过可直接利用的漏洞。
- **修复建议**：记录必要安全事件和上下文，但避免记录敏感数据。

### 速率限制、滥用防护或幂等保护不足
- **何时报告**：登录、验证码、重置密码、发券、下单、导出等高风险操作容易被暴力尝试、重放或刷接口时。
- **为什么重要**：属于实际可利用的设计风险，但要结合场景判断。
- **修复建议**：增加速率限制、重试限制、幂等键、二次确认或风控策略。

### 默认配置、调试开关或错误暴露不当
- **何时报告**：调试模式在生产链路开启、目录列表暴露、默认口令未移除、内部错误直接返回客户端时。
- **为什么重要**：会放大攻击面，但需要结合部署语境判断其严重度。
- **修复建议**：收紧生产配置，关闭调试输出，隐藏内部实现细节。

### 第三方依赖或外部组件风险信号明显
- **何时报告**：当前改动引入已知高危依赖、禁用了签名 / 校验、或把不可信第三方脚本直接接入关键链路时。
- **为什么重要**：供应链问题影响大，但需要与当前改动直接相关。
- **修复建议**：替换或升级高风险依赖，恢复完整性校验和来源约束。

## 5. 低优先级问题（谨慎报告）

### 通用安全增强建议
- **何时报告**：仅当项目规范明确要求，或它与当前改动直接相关且能明显降低实际风险时。
- **为什么重要**：像“建议启用更多安全头”“建议加强密码复杂度”这类内容，默认不应压过真实漏洞。

### 文档、命名或注释层面的安全语义不清
- **何时报告**：仅当权限边界、敏感字段或危险 API 的使用语义不清，已影响审查和后续修改安全性时。
- **为什么重要**：属于维护性问题，不应机械上报。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非已造成明确问题：

- 纯理论上的“可能有安全风险”，但没有可利用路径或证据支撑
- 与当前改动无关的历史安全债
- 泛泛要求“上 WAF / 上 MFA / 上零信任”之类的治理建议
- 没有部署上下文就断言配置一定不安全
- 把产品策略、运维基线或组织流程问题直接当作代码漏洞
- 可由独立安全扫描 / 资产盘点完成、但无法从当前 diff 高置信度判断的问题

## 7. 安全审查特有的判断重点

### 先看信任边界，再看实现细节
- 用户输入从哪里进来
- 在哪里被解释、渲染、执行、持久化或转发
- 中间有没有做校验、授权、净化和约束

### 优先报告“可利用且可修复”的问题
- 能说明攻击路径
- 能定位具体入口
- 能给出明确修复方向
- 比泛泛的“建议加强安全性”更有价值

### 区分代码漏洞与治理建议
- 代码审查应优先指出当前改动中的可利用风险
- 治理项、制度项、平台项可以提，但默认降级

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- 注入类问题
- XSS / HTML 注入
- 对象级授权缺失
- SSRF、路径遍历、文件上传边界错误
- 反序列化 / 动态执行风险
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 一般性的日志与告警补充
- 通用配置收紧建议
- 治理级别的安全增强建议
- 需要更大范围系统设计配合的问题

## 9. 审核输出建议

- 优先报告少量高价值、可利用、与当前改动直接相关的问题
- 报告安全问题时尽量说明攻击路径或利用前提，不要只贴名词
- 如果证据不足，应明确写成风险提示，而不是断言为漏洞
- 不要把 OWASP 条目逐条映射成评论清单
- 如果没有发现值得报告的高置信度安全问题，应明确说明

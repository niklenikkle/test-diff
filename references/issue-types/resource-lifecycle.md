# 资源与生命周期检查

本参考文档用于帮助判断：资源泄漏、生命周期失控、悬挂引用、清理缺失等问题是否已经有充分证据成立。

## 核心原则

不要把“大对象存在”“对象创建较多”“占用可能偏高”直接等同于泄漏。更关键的是：

- 是否存在明确的资源创建与持有
- 是否存在明确的清理、解绑、关闭、释放、回收边界
- 生命周期是否被外部引用异常拉长
- 失败、取消、中断、销毁路径是否同样安全

对文件句柄、连接、事务、订阅、定时器、后台任务等资源问题，优先判断的是“边界是否真的失守”，而不是“实现姿势是否完美”。如果只能说明写法保守性不足、错误处理不够完整或 Close / cleanup 姿势不理想，但还不能证明泄漏、数据丢失、句柄失控或生命周期被异常拉长，则默认不要升级为正式缺陷。

## 重点对象

- 监听器、订阅、事件总线回调
- 定时器、后台任务、协程、线程
- 文件句柄、网络连接、数据库连接、事务
- 缓存句柄、上下文对象、大对象、长生命周期对象
- UI 生命周期绑定资源

## 判定建议

以下情况更适合报为正式问题：
- 创建后缺少对应清理路径
- 销毁、失败或取消路径中资源未释放
- 外部引用、监听闭包、回调链导致对象生命周期被异常拉长
- 释放后仍可能继续访问或重复释放
- 能较直接说明这会导致资源泄漏、句柄耗尽、数据未落盘、事务未收敛或生命周期失控

以下情况更适合报为风险提示：
- 只能观察到大对象创建，但尚看不到外部持有链或生命周期异常
- 只能推测占用偏高，但尚无法说明资源边界缺失
- 只能看到 Close / cleanup 写法不够稳妥，但还不能稳定推出真实后果

## 输出建议

描述这类问题时，优先写清：
1. 资源是什么
2. 在哪里创建 / 注册 / 持有
3. 本应在哪里清理
4. 当前缺少的生命周期控制是什么
5. 为什么这会导致泄漏、悬挂引用或资源失控

若多个现象都来自同一资源边界缺失，优先合并为一条主问题，在影响分析中展开，不要把“未关闭”“可能泄漏”“失败路径不安全”等拆成多条重复正式问题。

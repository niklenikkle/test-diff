# 误报抑制与扩报收敛

本参考文档用于帮助在正式审查中控制噪音，避免把上下文现象、样式偏好或低置信度猜测扩展成独立正式问题。

## 核心原则

- 上下文文件首先用于解释当前改动，不应默认扩展为新的问题来源
- 只有当证据足以支撑独立风险结论时，才单独成项
- 低置信度现象优先写成风险提示，而不是确定性缺陷
- 不要因为能写更多，就主动扩报
- 若多个问题共享同一直接根因、同一执行路径或同一边界失守，默认合并为一条主问题，不要拆成多个标题相近的正式问题
- 对修复型提交，先判断本次改动是否在降低风险；默认压制的是残余争议点和“仍不够完美”的实现差异，而不是独立成立的新问题。若新改动引入了可直接证明的回归，或引入了虽未到 high 但仍值得现在处理的独立问题，仍可正式成项
- 未改动上下文里的历史问题、一般性最佳实践或可讨论风险，只有在被当前 diff 直接引入、放大、暴露或重新激活时，才正式成项

## 常见易误报场景

- 占位实现、空实现、TODO、本地 stub，但当前无法证明它在本次路径中形成真实缺陷
- 字段声明、配置扩展、结构调整，但风险主要仍在主变更路径
- 一般性大对象、缓存、循环、异步调用，但尚无证据证明其已造成泄漏、竞争或性能问题
- 纯粹基于“更优雅写法”或习惯偏好的建议
- 修复型提交中仍不够完美、但整体已更安全 / 更稳的实现差异
- 从同一主问题自然派生出的多个后果描述，例如同一次边界破坏带来的多种业务后果

## 成项建议

以下情况更适合单独成项：
- 与当前改动有直接因果关系
- 能说明成立条件、影响路径和修复方向
- 即使脱离主问题单独看，也属于值得作者现在处理的问题
- 与同文件其他问题相比，具备不同根因或需要不同修复动作

以下情况更适合并入主问题说明或直接省略：
- 只是主问题的背景噪音
- 证据不足以形成独立结论
- 只是在风格、抽象层次或可读性上存在轻微偏好差异
- 只是同一主问题的影响展开、后果拆分或风险放大表述
- 只是顺手看到的上下文风险，但与当前 diff 没有直接因果关系

## 输出建议

- 对低置信度现象，优先使用“风险提示”“需确认”“若前提成立则存在风险”等措辞
- 不要把同一证据链拆成多个重复问题，造成问题数虚高
- 不要让中低置信度问题压过真正高价值问题的排序
- 对修复型提交，若最终决定成项，应明确写出“新改动具体引入了什么回归”，而不是泛泛地说“仍有风险”

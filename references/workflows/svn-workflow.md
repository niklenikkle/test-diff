# SVN 审核流程 Reference

## 1. 适用范围

本参考适用于 `code-reviewer-optimized` 在 SVN 仓库中进行差异审查时使用，典型场景包括：

- 用户明确要求审查某个 SVN revision 或 revision 范围
- 用户直接提供 `svn-url`，要求围绕该 URL 对应路径做 diff 审查
- 当前工作副本只有 SVN 元信息，没有 Git PR / commit 上下文
- 需要根据 `svn diff`、`svn log`、`svn cat` 获取最小必要上下文完成 review
- 需要在批量提交、分支合并或历史版本差异中保持低噪音审查

本文件的目标不是教授 SVN 基础知识，而是帮助审查者在 SVN 环境下高效收集上下文并给出高置信度结论。

## 2. 审查目标

在 SVN 场景下做代码审查时，优先目标是：

- 明确当前审查的 revision 范围和涉及文件
- 用最小必要命令拿到足够上下文，而不是无边界扫描仓库
- 优先阅读 diff 与直接相关代码，避免因为上下文不足而过度猜测
- 在缺少 PR 描述、行内评论和讨论线程时，仍然保持审查意见收敛、具体、可执行
- 如果上下文不完整，明确说明判断边界

## 3. 推荐最小工作流

默认按以下顺序收集上下文。

### 第零步：先识别输入类型
先确认当前审查目标属于哪一种：

- 本地 working copy
- 单个 revision
- revision 范围
- 用户直接提供的 `svn-url`
- 用户直接提供的 SVN 提交文件（如包含 `svn_commit` / `file_diffs` 的 JSON）

如果用户提供的是 `svn-url`，则后续所有 `svn diff`、`svn log`、`svn cat`、`svn info` 都应围绕该 URL 执行，不要偷换成当前目录 working copy。

如果用户提供的是 SVN 提交文件，则应以该文件中的 `svn_commit`、`changed_files`、`file_diffs` 作为本次差异审查的权威输入；除非用户额外要求或文件明显不完整，否则不要再改用 working copy 或 `svn-url` 重新拼接上下文。

### 第一步：先看变更摘要
先确认这次审查到底改了哪些文件、改动范围有多大。

常用命令：

```bash
svn diff --summarize
svn diff --summarize -r START:END
svn diff --summarize -c REV
```

使用目的：

- 快速识别新增、修改、删除文件
- 判断是否存在无关改动混入
- 判断是否需要按文件分批审查

### 第二步：再看详细 diff
先基于摘要确认的范围阅读详细 diff，理解这次改动到底改了什么行为。

常用命令：

```bash
svn diff
svn diff -r START:END
svn diff -c REV
svn diff -r START:END path/to/file
```

使用目的：

- 理解行为变化而不是只看文件名
- 判断问题是否真的由当前改动引入
- 识别哪些文件需要继续补上下文

### URL 场景的最小工作流
如果用户直接提供 `svn-url`，默认按以下顺序收集上下文：

```bash
svn info URL
svn log URL -l 1 -v
svn diff --summarize -c REV URL
svn diff -c REV URL
svn cat -r REV URL_OR_FILE
```

其中：

- 若用户已明确给出 `REV` 或 `START:END`，则直接围绕该 URL + revision / range 获取摘要和 diff
- 若用户只给 `svn-url`、未给 revision，则必须先通过该 URL 的实际 `svn log -l 1 -v` 结果确定最近一次可见提交，再围绕该提交获取 diff
- “最近一次提交”必须以该 URL 对应路径的实际可见历史为准，不能偷换成整个仓库的最新 revision
- URL 可以指向仓库根、分支目录、子路径甚至单文件；审查范围应以该 URL 对应路径为边界，不要无边界扩展到其他路径
- 如果该 URL 无权限访问、网络不可达，或服务端不允许读取日志 / diff，应明确说明上下文获取失败，而不是假装完成审查
- 若需要生成正式审核报告，产物应写入统一审核输出根目录，而不是默认写回目标仓库目录；`svn-url` 场景可完全脱离本地 working copy

### 第三步：补提交元信息
在 SVN 没有 PR 描述时，`svn log` 往往是少数可用背景来源。

常用命令：

```bash
svn log -r START:END -v
svn log -r REV -v
```

使用目的：

- 确认提交说明、作者、路径范围
- 对照“提交意图”和“实际改动”是否一致
- 判断该 revision 是否可能是合并提交、批量修复或联动改动

### 第四步：仅在必要时补文件上下文
如果 diff 不足以判断问题，再看前后版本文件内容。

常用命令：

```bash
svn cat -r START path/to/file
svn cat -r END path/to/file
svn cat -r REV path/to/file
```

使用目的：

- 对比变更前后完整逻辑
- 理解删除代码、重构替换、接口签名变化
- 验证问题是否属于当前修改而不是历史遗留

### 何时优先用 URL 模式而不是 working copy
- 用户消息里直接给了 `svn-url`
- 本地没有 working copy，或当前目录与目标仓库无关
- 用户明确要求审查某个仓库路径、分支目录或子路径，而不是当前检出内容

### URL 模式下的额外边界要求
- 如果 URL 指向子目录，则不要把父目录或兄弟目录的提交混进审查范围
- 如果 URL 指向单文件，则重点围绕该文件及其本次 diff 做判断，只有在证据需要时才补相邻上下文
- 如果 `svn log URL -l 1 -v` 返回的是路径受限视角下的最近提交，应接受这一边界，不要再自行扩大到全仓最新 revision

## 4. 获取上下文的优先顺序

默认遵循以下优先级：

1. 先看变更文件列表
2. 再看 diff 本身
3. 再看关键函数、类、模块上下文
4. 最后才看完整文件
5. 除非用户明确要求，不做全仓扫描

这条顺序很重要。SVN 场景比 Git PR 更容易缺少讨论上下文，因此更要控制阅读边界，避免把不相关历史问题带入当前 review。

## 5. 在 SVN 审查里优先关注什么

### 审查范围是否清楚
- 当前是工作副本 diff、单个 revision，还是 revision 区间
- 是否存在多个不相干功能被打包进同一范围
- 是否需要按文件或按模块拆开判断

### 提交说明与实际改动是否一致
- 提交说明说是“小修复”，但 diff 实际包含结构性改动
- 提交说明说是“重构”，但行为逻辑也一起变了
- 提交说明缺少关键信息，导致需要更谨慎地控制结论

### 是否缺少配套改动
- 接口签名变了，但调用方没同步
- 数据结构变了，但序列化 / 反序列化或持久层没同步
- 权限逻辑改了，但测试和边界处理没跟上
- 配置 / schema / 文档 / 测试是否需要联动更新

### 是否混入无关改动
- 同一 revision 中包含格式化、重命名、逻辑修改混杂
- 合并提交把大量无关文件带进来
- 审查意见应优先聚焦风险最高、最直接相关的改动

### 是否需要回看前一版逻辑
以下情况通常值得用 `svn cat` 补上下文：

- 大段删除 / 替换，无法只从 diff 判断语义变化
- 条件分支调整，需确认原行为是否被破坏
- 重构后函数名没变，但调用时机或副作用可能已改变
- 接口 / schema / 协议变更，需要比对前后版本约定

## 6. 不要轻易做什么

以下做法在 SVN 审查里应默认避免：

- 不要因为缺少 Git PR 上下文就脑补需求和业务目标
- 不要一上来扫描整个仓库或整段长期历史
- 不要把 revision 范围外的历史问题当成这次审查的主意见
- 不要把 SVN 分支、目录结构、版本控制常识写成代码审查意见
- 不要因为一次提交很大，就泛泛而谈地给出笼统评论
- 不要在上下文不够时把猜测写成确定结论

如果 revision 范围过大、提交混杂严重、上下文不足，应明确说明“当前判断基于已有版本范围”，必要时建议拆分审查。

## 7. 输出建议

在 SVN 场景下，输出建议尽量包含以下信息：

### 先说明审查范围
- 审查的是工作副本 diff，还是 `r1234:r1235`
- 重点关注了哪些文件或模块
- 是否存在上下文不足或提交过大的限制

### 再给结论
- 是否发现值得报告的高置信度问题
- 如果没有，应明确说明
- 如果有，应按风险和确定性排序

### 问题描述尽量贴近当前 revision
- 优先指出“当前变更引入了什么问题”
- 避免把整段历史设计债笼统塞进本次 review
- 如果问题依赖前后版本对比才能成立，应简要说明比较依据

### 必要时说明判断边界
- “基于 r1234:r1235 的 diff 判断”
- “当前未展开全文件 / 全仓扫描”
- “该问题是否成立取决于某个调用方是否同步更新”

## 8. 何时建议拆分或收敛审查

以下情况不适合一次性给出大而泛的 review，应优先收敛：

- revision 范围过大，包含多个独立功能
- 合并提交混入大量机械改动
- 重命名 / 格式化与逻辑修改混杂
- 关键上下文不在当前范围内，继续扩展会失控

这时更好的做法是：

1. 先聚焦最关键文件
2. 优先报告高置信度问题
3. 明确指出其余部分若需继续审查，建议拆分范围

## 9. 常用命令附录

只保留 SVN 审查真正常用的命令。

```bash
# 查看当前工作副本状态
svn status

# 查看变更摘要
svn diff --summarize
svn diff --summarize -r START:END
svn diff --summarize -c REV

# 查看详细 diff
svn diff
svn diff -r START:END
svn diff -c REV
svn diff -r START:END path/to/file

# 查看提交日志和变更路径
svn log -r REV -v
svn log -r START:END -v

# 查看某个 revision 下的文件内容
svn cat -r REV path/to/file
svn cat -r START path/to/file
svn cat -r END path/to/file

# 查看工作副本信息
svn info
```

## 10. 使用原则总结

- 先摘要，后 diff，再补必要上下文
- 优先围绕当前 revision 范围做判断
- 不因为 SVN 缺少 PR 上下文就扩大阅读边界
- 不把版本控制知识当作审查内容
- 目标是做低噪音、高置信度、与当前改动直接相关的 review

# Rust 审核标准

## 1. 适用范围

本参考适用于 Rust 代码审核，重点覆盖：

- 所有权 / 借用 / 生命周期相关逻辑
- `unsafe`、FFI 边界、并发、错误处理
- 内存安全、整数溢出意图、资源管理、性能关键路径

当变更涉及 `unsafe`、裸指针、FFI、并发、复杂生命周期、错误传播或零拷贝优化时，应优先参考本文件。

## 2. 审核目标

审核 Rust 代码时，优先关注：

- 是否在绕过编译器安全保证的地方引入未定义行为风险
- 是否使用 `unwrap` / `expect` 等让关键路径在运行时 panic
- 是否在 FFI、并发、整数运算、裸指针使用中缺少边界保护
- 是否为了性能过早复杂化实现，增加维护成本和错误风险

## 3. 高优先级问题（优先报告）

### `unsafe` 使用缺少必要边界说明或前置校验
- **何时报告**：`unsafe` 块缺少充分理由、安全前提不清晰，或依赖调用者保证但未文档化时。
- **为什么重要**：会绕过 Rust 的核心安全保障。
- **建议如何审**：关注裸指针解引用、切片构造、FFI、手写内存管理、并发共享。
- **修复建议**：最小化 `unsafe` 范围，补充安全注释，先在进入 `unsafe` 前校验前置条件。
- **何时给代码示例**：建议给。

### FFI 输入未校验
- **何时报告**：来自 C / 外部调用的数据指针、长度、字符串、句柄未检查就直接使用时。
- **为什么重要**：容易导致越界、空指针解引用、未定义行为。
- **修复建议**：检查空指针、长度、编码与所有权边界，必要时返回错误码。
- **何时给代码示例**：建议给。

### 关键路径滥用 `unwrap` / `expect`
- **何时报告**：`unwrap` / `expect` 出现在用户输入、I/O、网络、解析、外部依赖结果等不可靠路径时。
- **为什么重要**：会把可恢复错误变成运行时崩溃。
- **修复建议**：改为 `?`、`match`、`ok_or`、自定义错误传播。
- **何时给代码示例**：建议给。

### 并发共享状态不安全
- **何时报告**：通过 `Arc<Mutex<_>>`、`unsafe impl Send/Sync`、跨线程共享可变状态时存在死锁、锁范围过大或线程安全假设不成立时。
- **为什么重要**：会引发死锁、数据竞争或逻辑阻塞。
- **修复建议**：缩小锁范围，避免不必要共享，重新审视 Send/Sync 边界。
- **何时给代码示例**：建议给。

### 整数溢出意图不明确
- **何时报告**：数值运算可能溢出，但代码未明确说明应使用 checked / saturating / wrapping 何种语义时。
- **为什么重要**：debug 和 release 行为可能不同，容易造成隐藏 bug。
- **修复建议**：根据业务语义显式选择对应运算方式。
- **何时给代码示例**：视复杂度而定，常建议给。

## 4. 中优先级问题（按上下文报告）

### 不必要的克隆或所有权移动复杂化
- **何时报告**：为绕过借用检查引入大量 clone，明显增加成本或掩盖设计问题时。
- **为什么重要**：会损害性能并降低可读性。
- **修复建议**：重构函数签名与借用关系，必要时使用引用或切片。

### 错误类型设计过于宽泛
- **何时报告**：大量 `Box<dyn Error>`、字符串错误或上下文不足导致调用方难以处理时。
- **为什么重要**：会降低错误传播质量。
- **修复建议**：使用明确错误枚举或错误包装方案。

### 生命周期 / 泛型复杂度明显超出收益
- **何时报告**：实现因追求零拷贝或泛型抽象而明显增加维护风险时。
- **为什么重要**：并非所有场景都值得换取复杂度。
- **修复建议**：在清晰性和性能之间重新平衡。

## 5. 低优先级问题（谨慎报告）

### 命名、模块组织、导出粒度可优化
- **何时报告**：仅当其显著影响理解或与项目约定冲突时。

### 轻度迭代器风格建议
- **何时报告**：仅当当前写法已明显影响可读性或性能时。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见：

- 单纯因为“还能更零拷贝”就强行复杂化代码
- 仅从风格偏好出发要求 iterator / match / combinator 改写
- 没有实际风险时对局部 clone 过度上纲上线
- 与当前改动无关的历史模块组织问题

## 7. 该语言特有的审查重点

### 安全边界
- `unsafe`、FFI、裸指针、跨线程共享是 Rust 审核的最高风险区域

### 错误处理
- `unwrap` / `expect` 是否出现在不可靠路径上，是非常高价值的审查点

### 性能与复杂度平衡
- Rust 中很多“优化”会引入额外复杂度，审查时要评估收益是否真实成立

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- `unsafe` 边界问题
- FFI 输入校验问题
- `unwrap` / `expect` 导致的 panic 风险
- 并发共享状态问题
- 用户明确要求给出改法

以下情况通常只给修改方向即可：

- 局部 clone 优化
- 模块结构建议
- 轻度泛型 / 生命周期整理

## 9. 审核输出建议

- 优先报告安全边界、panic 风险、并发与 FFI 问题
- 不要把“更 Rust 风格”当作默认高价值意见
- 如果提出性能建议，应说明它是否真的影响当前关键路径
- 若未发现值得报告的高置信度问题，应明确说明

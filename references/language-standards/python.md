# Python 审核标准

## 1. 适用范围

本参考适用于 Python 代码审核，重点覆盖：

- 后端服务、脚本、CLI、自动化任务
- 数据处理、文件处理、外部服务调用、数据库访问
- 输入边界、异常处理、资源管理、并发与反序列化风险

当变更涉及用户输入、文件系统、数据库、序列化、网络请求、异常处理、并发、批处理逻辑时，应优先参考本文件。

## 2. 审核目标

审核 Python 代码时，优先关注：

- 是否存在边界条件、默认值、异常流导致的逻辑错误
- 是否存在 `eval` / `exec` / `pickle` / 路径遍历 / SQL 注入等明确安全风险
- 是否存在资源泄漏、异常吞掉、错误恢复路径缺失
- 是否存在会明显影响稳定性或可维护性的复杂度问题
- 是否在脚本便利性和生产可靠性之间失衡

## 3. 高优先级问题（优先报告）

### 可变默认参数导致状态污染
- **何时报告**：函数参数默认值使用 `[]`、`{}`、`set()` 等可变对象，且函数会修改它们时。
- **为什么重要**：会在多次调用之间共享状态，造成隐蔽 bug。
- **建议如何审**：重点看工具函数、收集器、递归函数、构造辅助函数。
- **修复建议**：改为 `None` 作为默认值，并在函数内部初始化。
- **何时给代码示例**：建议给。

### `eval` / `exec` / 不安全反序列化
- **何时报告**：使用 `eval`、`exec`、`pickle.loads`、`yaml.load`（不安全 loader）处理不可信输入时。
- **为什么重要**：可能导致任意代码执行或恶意对象注入。
- **建议如何审**：关注配置加载、任务编排、插件系统、动态表达式解析。
- **修复建议**：改用 `ast.literal_eval`、JSON、安全 loader 或显式白名单机制。
- **何时给代码示例**：安全问题，建议给。

### SQL 拼接或查询参数未绑定
- **何时报告**：用户输入通过 f-string、`format`、字符串拼接进入 SQL 时。
- **为什么重要**：存在 SQL 注入风险。
- **建议如何审**：关注 ORM 原生查询、报表脚本、动态过滤条件、排序和分页参数。
- **修复建议**：使用参数化查询，并对白名单字段做额外限制。
- **何时给代码示例**：建议给。

### 路径拼接直接使用用户输入
- **何时报告**：文件路径直接由用户输入、URL 参数、解压文件名、上传文件名拼接生成且未校验时。
- **为什么重要**：容易引发路径遍历、任意文件覆盖或读取。
- **建议如何审**：检查文件导入导出、模板渲染、压缩包解压、缓存文件生成。
- **修复建议**：使用 `pathlib` / `resolve()` 做基准目录校验，并限制文件名来源。
- **何时给代码示例**：建议给。

### 异常捕获过宽且吞掉错误
- **何时报告**：使用裸 `except:`、`except Exception:` 后直接 `pass`、返回默认值或记录不足以定位问题的信息时。
- **为什么重要**：会掩盖真实故障，制造静默失败。
- **建议如何审**：重点看任务调度、批处理、网络重试、解析逻辑、后台 worker。
- **修复建议**：收窄异常范围，记录上下文，决定是否继续抛出。
- **何时给代码示例**：建议给，尤其是异常流复杂时。

### 资源管理不当
- **何时报告**：文件、socket、数据库连接、锁、线程池、临时资源未正确关闭或释放时。
- **为什么重要**：会导致句柄泄漏、连接耗尽、数据不一致。
- **建议如何审**：关注 `open()`、requests 会话、数据库连接、线程池 / 进程池、临时文件。
- **修复建议**：优先使用上下文管理器 `with`，或显式 finally 清理。
- **何时给代码示例**：建议给。

### 不可信输入未经校验进入关键逻辑
- **何时报告**：请求参数、CLI 参数、配置、环境变量、CSV/JSON 字段未经校验就决定查询、路径、权限或任务执行逻辑时。
- **为什么重要**：会造成脏数据、逻辑绕过或安全问题。
- **建议如何审**：查看 API 边界、任务入口、配置加载和数据清洗逻辑。
- **修复建议**：在入口增加 schema 校验、类型转换和白名单约束。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### 边界条件和空值处理不完整
- **何时报告**：空列表、缺失键、None、超时、部分成功、重试失败等分支未被正确处理时。
- **为什么重要**：Python 容易在“看起来能跑”的情况下留下边缘 bug。
- **修复建议**：补充显式分支、默认值策略或错误路径。

### 并发或共享状态风险
- **何时报告**：多线程 / 多进程 / 异步任务共享可变状态、缺少同步或取消机制时。
- **为什么重要**：会带来数据竞争、重复处理、资源泄漏。
- **修复建议**：隔离共享状态，使用锁、队列、上下文或幂等保护。

### 函数签名、返回语义或模块边界变更未同步
- **何时报告**：服务层、任务函数、CLI 入口、工具函数的参数顺序、默认值、返回结构、异常语义发生变化后，调用点、包装层或上下游模块没有同步更新时；或跨层导入破坏既有分层约定时。
- **为什么重要**：Python 动态性强，这类问题容易绕过静态检查，最后在运行时才暴露为隐蔽回归。
- **修复建议**：同步更新调用点与适配层，减少跨层直连，必要时补充契约测试或边界断言。

### 日志记录不足或泄露敏感信息
- **何时报告**：错误时缺少必要上下文，或把 token、密码、完整请求体写入日志时。
- **为什么重要**：前者影响排障，后者带来泄露风险。
- **修复建议**：记录必要上下文但脱敏敏感字段。

### 魔法数字或业务常量散落
- **何时报告**：分页大小、重试次数、状态阈值、业务规则常量在多个位置硬编码，已经影响理解、统一修改或边界判断时。
- **为什么重要**：这类问题不一定直接造成 bug，但会明显增加后续修改时的漏改风险。
- **修复建议**：提取具名常量，集中配置，并让调用处表达业务语义。

### 明显低效的数据处理路径
- **何时报告**：大文件一次性读入、大量重复查询、热点循环中重复计算、列表成员检查使用线性结构导致明显性能损耗时。
- **为什么重要**：在批处理和数据脚本里影响尤其明显。
- **修复建议**：改为流式处理、批量查询、缓存或更合适的数据结构。

## 5. 低优先级问题（谨慎报告）

### 类型注解缺失
- **何时报告**：仅当项目明确依赖类型检查，或缺失类型已明显影响接口理解和改动安全性时。
- **为什么重要**：它是维护性增强项，不应默认压过真实 bug。

### 重复的边界 / 权限 / 校验逻辑
- **何时报告**：同类视图、服务或脚本中重复实现校验、权限、事务、异常处理、序列化逻辑，已经出现明显分叉风险时。
- **为什么重要**：重复逻辑会放大漏改、修复不一致和未来回归风险。
- **修复建议**：抽取共享函数、基类、装饰器或公共校验入口，收敛边界处理。

### Docstring 或注释不足
- **何时报告**：仅当导出 API、复杂算法、边界行为难以理解，且项目规范对此有要求时。
- **为什么重要**：属于可维护性问题，不应机械上报。

### 推导式、生成器、风格优化机会
- **何时报告**：仅当当前写法已经明显影响内存、可读性或效率时。
- **为什么重要**：不要把“更 Pythonic”本身当作高价值审查意见。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非已造成明确问题：

- 单纯因为代码不够“Pythonic”就要求改写
- 机械要求所有函数都加类型注解
- 机械要求所有函数都写 docstring
- 把列表推导式 / 字典推导式当成默认更优解强行上报
- 不影响行为的导入顺序、命名风格细节
- 与当前变更无关的历史代码风格问题

## 7. 该语言特有的审查重点

### Python 语言特性
- 动态类型和运行时行为灵活，边界错误和错误路径更容易漏掉
- 默认参数、可变对象、鸭子类型和异常处理是高风险区域

### 常见高价值风险
- `eval` / `exec`
- `pickle.loads` 等不安全反序列化
- broad except 导致静默失败
- 路径拼接与文件处理缺少校验
- 数据处理代码对空值、缺失字段、格式偏差过于乐观

### 后端 / 脚本场景
- API 边界输入校验、数据库参数绑定、外部请求超时、重试策略、日志脱敏、资源关闭是重点审核项

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- 可变默认参数
- `eval` / `exec` / `pickle` / SQL 注入 / 路径遍历等安全问题
- broad except 或异常流复杂问题
- 资源管理与上下文管理器使用不当
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 类型注解增强
- docstring 补充
- 轻度结构整理
- 一般性 Pythonic 写法优化

## 9. 审核输出建议

- 优先报告安全、正确性、异常流、资源管理方面的问题
- 对风格、注释、类型提示类建议默认降级
- 如果提出性能问题，应说明其是否会在当前数据规模或调用路径下产生实际影响
- 如果提出可维护性问题，应说明它如何增加未来修改风险
- 在正式报告或仓库级扫描中，对**明确成立**的日志缺失、魔法数字、重复边界逻辑，可以作为中低优问题正式输出，不要因为它们不阻断合并就完全省略
- 如果没有发现值得报告的高置信度问题，应明确说明

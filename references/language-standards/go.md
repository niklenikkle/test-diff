# Go 审核标准

## 1. 适用范围

本参考适用于 Go 代码审核，重点覆盖：

- HTTP 服务、RPC 服务、后台任务、并发处理逻辑
- goroutine、channel、context、锁、资源管理
- 数据库访问、输入验证、错误处理、外部服务调用

当变更涉及并发、取消机制、共享状态、数据库、网络请求、文件处理或长生命周期任务时，应优先参考本文件。

## 2. 审核目标

审核 Go 代码时，优先关注：

- 是否存在 goroutine 泄漏、死锁、竞态条件
- 是否存在 SQL 注入、输入未校验、敏感信息泄露等安全问题
- 是否正确处理错误、超时、取消和资源释放
- 是否引入明显的切片 / map / 字符串 / I/O 性能问题
- 是否与项目的接口、并发、错误处理模式保持一致

## 3. 高优先级问题（优先报告）

### goroutine 泄漏
- **何时报告**：后台 goroutine 缺少退出条件、未响应 `context.Done()`、依赖永不关闭的 channel 或阻塞发送/接收时。
- **为什么重要**：会导致资源泄漏、任务堆积、服务长期不稳定。
- **建议如何审**：关注 worker、订阅、轮询、异步回调、重试任务。
- **修复建议**：引入 `context`、明确关闭通道、增加退出分支。
- **何时给代码示例**：建议给。

### channel 死锁或关闭语义错误
- **何时报告**：无接收方发送、重复关闭 channel、向已关闭 channel 发送、关闭责任不清时。
- **为什么重要**：会直接 panic 或卡死流程。
- **建议如何审**：查看生产者 / 消费者边界、channel 所有权、缓冲大小与同步逻辑。
- **修复建议**：明确“谁发送、谁关闭”，拆清所有权与同步职责。
- **何时给代码示例**：建议给。

### 共享状态竞态条件
- **何时报告**：多个 goroutine 读写共享变量，缺少锁、原子操作或消息传递隔离时。
- **为什么重要**：会导致结果不一致、数据损坏、线上偶发 bug。
- **建议如何审**：关注计数器、缓存、状态机、连接池、全局 map。
- **修复建议**：使用 `sync.Mutex`、`sync.RWMutex`、`atomic` 或 channel 模型隔离共享状态。
- **何时给代码示例**：建议给。

### 错误被忽略或吞掉
- **何时报告**：使用 `_` 丢弃错误、记录不足、返回值包装不清晰导致调用方无法判断失败原因时。
- **为什么重要**：Go 的可靠性高度依赖显式错误处理。
- **建议如何审**：查看 I/O、网络、解析、数据库、并发任务收尾路径。
- **修复建议**：显式处理错误，补充上下文，必要时使用 `%w` 包装。
- **何时给代码示例**：建议给。

### 外部请求缺少超时或取消机制
- **何时报告**：HTTP / RPC / DB / 阻塞操作缺少超时与 context 传播时。
- **为什么重要**：容易造成请求堆积和资源耗尽。
- **建议如何审**：关注 `http.Client`、数据库调用、第三方 SDK、后台任务。
- **修复建议**：引入 context deadline / timeout，并确保向下游传递。
- **何时给代码示例**：建议给。

### SQL 拼接或输入未校验
- **何时报告**：用户输入经 `fmt.Sprintf`、字符串拼接进入 SQL，或关键业务参数未校验时。
- **为什么重要**：存在注入与脏数据风险。
- **修复建议**：参数化查询、输入白名单 / schema 校验。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### 切片 / map 未预分配导致明显开销
- **何时报告**：热点路径中已知容量却频繁扩容时。
- **为什么重要**：会放大 GC 和分配成本。
- **修复建议**：按需预分配容量。

### 字符串或字节处理低效
- **何时报告**：循环中反复拼接字符串、频繁 `string` / `[]byte` 转换导致明显成本时。
- **为什么重要**：在日志、序列化、文本处理路径中影响明显。
- **修复建议**：使用 `strings.Builder`、减少转换。

### 资源关闭责任不清
- **何时报告**：文件、响应体、rows、ticker、timer 等关闭 / stop 时机混乱时。
- **为什么重要**：容易造成泄漏或重复关闭问题。
- **修复建议**：显式约定资源生命周期，必要时用 `defer` 且注意作用域。

### 函数签名、接口约定或错误语义变更未同步
- **何时报告**：函数、方法、接口的参数顺序、零值语义、返回结构、错误包装方式或 sentinel error 语义发生变化后，调用点、适配层或上下游包没有同步更新时；或跨 package 直接依赖破坏既有分层约定时。
- **为什么重要**：Go 很多回归不会表现为编译错误，而是体现在错误判断失效、零值误用或运行时行为悄悄变化。
- **修复建议**：同步更新调用方和适配层，减少跨包直连，必要时补充契约测试、错误断言和边界用例。

## 5. 低优先级问题（谨慎报告）

### 命名或接口形式可优化
- **何时报告**：仅当其明显影响理解或违背项目约定时。

### 可读性层面的轻度重构机会
- **何时报告**：仅当函数职责明显过重、错误路径难以理解时。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见：

- 纯粹因为“不够 idiomatic Go”就强行要求改写
- 没有实际收益时机械要求每处都用接口抽象
- 对小型局部代码强行做性能微优化
- 与当前改动无关的历史命名或包结构问题

## 7. 该语言特有的审查重点

### 并发模型
- goroutine 生命周期、channel 所有权、context 传播是 Go 审核中的最高价值区域之一

### 错误处理
- Go 倾向显式返回错误，忽略错误通常不是小问题

### 资源与性能
- 切片 / map 分配、I/O、字符串处理、资源关闭在服务端代码中很常见

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- goroutine 泄漏
- channel 死锁 / 关闭错误
- 竞态条件
- 错误处理缺失
- 超时 / context 传播缺失
- 用户明确要求给出改法

以下情况通常只给修改方向即可：

- 轻度命名优化
- 一般性预分配建议
- 小型结构整理

## 9. 审核输出建议

- 优先报告并发、错误处理、安全和资源生命周期问题
- 只有在性能影响明确时才报告微观优化问题
- 如果提出并发问题，应说明它会造成死锁、泄漏还是竞态
- 若未发现值得报告的高置信度问题，应明确说明

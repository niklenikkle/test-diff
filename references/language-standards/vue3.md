# Vue3 审核标准

## 1. 适用范围

本参考适用于 Vue3 代码审核，重点覆盖：

- Composition API、`script setup`、响应式数据与模板绑定
- 组件通信、props / emits、provide / inject、Pinia 等状态管理
- 生命周期、副作用清理、异步数据获取、列表渲染与性能优化
- 与输入渲染、权限边界、HTML 注入、请求状态相关的问题

当变更涉及响应式状态、组件边界、模板逻辑、生命周期、副作用、异步请求或大型列表时，应优先参考本文件。

## 2. 审核目标

审核 Vue3 代码时，优先关注：

- 是否存在响应式失效、状态不同步、模板逻辑错误
- 是否存在生命周期、副作用清理、异步竞态问题
- 是否存在 XSS、未验证输入、权限边界缺失等安全风险
- 是否引入明显的渲染性能问题或组件职责失衡
- 是否违背项目既有组件模式，增加维护成本

## 3. 高优先级问题（优先报告）

### 直接修改 props 或破坏单向数据流
- **何时报告**：子组件直接修改 props，或绕过 emit / 状态管理约定修改上游状态时。
- **为什么重要**：会导致数据流混乱、状态同步异常和维护困难。
- **建议如何审**：关注 `defineProps` 返回对象的使用、双向绑定实现、表单组件、包装组件。
- **修复建议**：改用 `emit('update:...')`、局部副本或集中状态管理。
- **何时给代码示例**：建议给。

### `reactive()` 解构导致响应性丢失
- **何时报告**：直接从 `reactive()` 对象解构字段并在后续逻辑或模板中使用，且未通过 `toRefs` / `toRef` 处理时。
- **为什么重要**：会造成视图不更新、状态错乱、隐蔽 bug。
- **建议如何审**：关注 setup 中的对象解构、组合式函数返回值处理、模板绑定字段来源。
- **修复建议**：使用 `toRefs` / `toRef`，或改为 `ref` 管理单值。
- **何时给代码示例**：建议给。

### 生命周期和副作用清理缺失
- **何时报告**：事件监听、定时器、订阅、请求取消、观察器等副作用未在卸载时清理时。
- **为什么重要**：会导致内存泄漏、重复监听、页面行为异常。
- **建议如何审**：关注 `onMounted`、`watch`、`watchEffect`、全局事件、第三方库接入。
- **修复建议**：使用 `onUnmounted` / 清理函数，必要时引入 abort / stop 机制。
- **何时给代码示例**：建议给。

### 模板中渲染不可信 HTML
- **何时报告**：使用 `v-html` 渲染用户输入、外部富文本或未净化内容时。
- **为什么重要**：存在明确 XSS 风险。
- **建议如何审**：关注 CMS 内容、评论区、Markdown 渲染、后台配置驱动内容。
- **修复建议**：优先纯文本渲染，或使用经过审计的净化流程。
- **何时给代码示例**：安全问题，建议给。

### 异步请求竞态导致状态覆盖
- **何时报告**：多个请求可能交错返回，旧请求覆盖新状态，或组件卸载后仍回写状态时。
- **为什么重要**：容易导致界面显示错误数据。
- **建议如何审**：关注搜索、筛选、分页、快速切换路由、watch 触发请求的逻辑。
- **修复建议**：增加取消机制、请求标识、最新响应判定或状态机式管理。
- **何时给代码示例**：建议给。

### 组件级权限或对象级授权缺失
- **何时报告**：仅在前端隐藏按钮而缺少服务端/数据层保护，或前端逻辑默认信任未经校验的资源访问时。
- **为什么重要**：容易形成越权路径。
- **建议如何审**：查看编辑、删除、管理操作、文件访问、组织资源页面。
- **修复建议**：强调权限校验必须落到服务端，并在前端补充合理的状态约束。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### `v-for` key 不稳定
- **何时报告**：对可重排、可增删列表使用索引作为 key，且可能导致状态错位时。
- **为什么重要**：会引发组件复用错误和 UI 状态异常。
- **修复建议**：使用稳定唯一标识作为 key。

### `v-if` 与 `v-for` 混用
- **何时报告**：在同一元素上同时使用 `v-if` 和 `v-for`，导致可读性差或渲染逻辑不稳定时。
- **为什么重要**：容易产生隐性性能问题和逻辑歧义。
- **修复建议**：通过 `computed` 先过滤，再渲染。

### 组件通信方式失衡
- **何时报告**：props/emits 过深、provide/inject 滥用、Pinia 使用边界不清，导致依赖关系模糊时。
- **为什么重要**：会抬高维护成本。
- **修复建议**：根据层级与共享范围重选通信方式。

### 计算属性与侦听器使用不当
- **何时报告**：把纯派生数据放进 `watch` 做副作用式计算，或在模板中反复调用昂贵方法时。
- **为什么重要**：会降低可读性并带来额外渲染成本。
- **修复建议**：纯派生数据优先 `computed`，副作用才使用 `watch`。

## 5. 低优先级问题（谨慎报告）

### props / emits 类型定义不够精确
- **何时报告**：仅当其已影响组件边界理解、误用概率高，或项目明确要求严格类型时。

### 组件拆分机会
- **何时报告**：当单文件组件已明显职责过多、模板逻辑拥挤、后续修改困难时。

### 一般性模板整理
- **何时报告**：仅在明显影响理解或违反项目规范时。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见：

- 单纯因为“可以拆组件”就强行要求重构
- 没有实际问题时机械要求所有逻辑都提成 composable
- 仅因写法不同就否定 props/emits 或 Pinia 方案
- 对小型模板进行纯风格层面的整理建议
- 与当前改动无关的历史组件结构问题

## 7. 该语言特有的审查重点

### Vue3 / Composition API
- 响应式对象解构、ref 自动解包、watch / computed 边界是高价值问题集中区
- 副作用管理依赖生命周期，清理缺失很常见

### 模板与渲染
- `v-html`、列表 key、条件渲染、事件绑定和表单双向绑定值得重点关注

### 状态与通信
- props / emits、provide / inject、Pinia 的边界是否清晰，是维护性关键点

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- props 误改
- 响应式丢失
- 生命周期清理缺失
- `v-html` 引入的安全问题
- 异步竞态问题
- 用户明确要求给出改法

以下情况通常只给修改方向即可：

- 组件拆分建议
- 轻度模板整理
- 一般性通信方式优化

## 9. 审核输出建议

- 优先报告响应式正确性、副作用管理和安全问题
- 不要把本文件当成“Vue3 最佳实践检查表”逐条打卡
- 若提出性能问题，应说明是在列表、热点渲染还是频繁交互场景下成立
- 若未发现值得报告的高置信度问题，应明确说明

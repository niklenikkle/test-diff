# PHP 审核标准

## 1. 适用范围

本参考适用于 PHP 代码审核，重点覆盖：

- Web 应用、后台管理、API、CLI、任务调度与传统服务端逻辑
- Laravel / Symfony / 原生 PHP 的控制器、服务层、ORM、模板与中间件代码
- 请求输入、文件上传、数据库访问、认证授权、会话 / Cookie、异常处理与缓存
- 与 SQL 注入、XSS、文件上传、反序列化、权限绕过和 N+1 查询相关的风险点

当变更涉及表单、模板输出、路由参数、数据库查询、文件处理、鉴权逻辑或框架中间件时，应优先参考本文件。

## 2. 审核目标

审核 PHP 代码时，优先关注：

- 是否存在输入边界错误、模板输出错误、异常流缺失或状态变更逻辑漏洞
- 是否存在 SQL 注入、XSS、文件上传、反序列化、命令执行等明确安全风险
- 是否因为框架便利性而忽略了授权边界、校验边界和错误处理
- 是否出现明显的 N+1、无分页查询、缓存失效或阻塞型低效处理
- 是否与项目既有框架约定、分层方式和异常处理风格冲突

## 3. 高优先级问题（优先报告）

### 输入未经校验直接进入查询、状态修改或权限判断
- **何时报告**：`$_GET` / `$_POST` / 路由参数 / JSON 输入 / 表单数据未经验证就用于查询、对象创建、状态更新或权限判断时。
- **为什么重要**：容易造成脏数据、逻辑绕过和业务漏洞。
- **建议如何审**：关注 controller、Form Request、DTO、验证器、枚举值、排序分页字段、布尔开关参数。
- **修复建议**：在入口增加显式验证和白名单约束，不要把“字段存在”当成“输入合法”。
- **何时给代码示例**：建议给。

### SQL 拼接或动态查询条件缺少约束
- **何时报告**：用户输入通过字符串拼接进入 SQL，或排序字段、列名、where 片段未做白名单限制时。
- **为什么重要**：存在 SQL 注入和越权数据读取风险。
- **建议如何审**：查看原生 SQL、Query Builder 动态条件、报表接口、搜索接口、导出功能。
- **修复建议**：使用参数绑定；对列名、排序方向等不能参数化的部分使用白名单映射。
- **何时给代码示例**：安全问题，建议给。

### 模板输出或富文本渲染存在 XSS 风险
- **何时报告**：用户输入、CMS 内容、URL 参数、富文本结果未经转义直接输出，或错误使用原样输出语法时。
- **为什么重要**：会直接导致前端脚本注入。
- **建议如何审**：关注 Blade / Twig 模板、`echo`、富文本组件、Markdown 渲染、自定义 helper。
- **修复建议**：默认使用安全转义输出；仅对可信内容使用原样输出，并在进入模板前完成净化。
- **何时给代码示例**：建议给。

### 文件上传、文件路径或反序列化处理不安全
- **何时报告**：上传文件类型、大小、扩展名、保存路径未校验，或直接使用 `unserialize()` / 用户可控路径处理文件时。
- **为什么重要**：会造成任意文件写入、路径遍历或对象注入风险。
- **建议如何审**：查看上传控制器、导入导出、临时文件处理、缓存反序列化、附件下载接口。
- **修复建议**：做 MIME / 扩展名 / 大小校验，使用安全文件名和固定目录；避免对不可信数据反序列化。
- **何时给代码示例**：安全问题，建议给。

### 授权只停留在页面或路由层
- **何时报告**：页面按钮隐藏、路由中间件通过后，具体资源操作未再次校验当前用户是否有权访问时。
- **为什么重要**：容易出现对象级越权。
- **建议如何审**：关注订单、用户、租户、后台资源、导出删除修改操作是否结合资源所有权做判断。
- **修复建议**：在资源读取和写入入口增加显式授权，前端和路由层只做第一层过滤。
- **何时给代码示例**：建议给。

### 异常被吞掉或错误处理泄露内部细节
- **何时报告**：`catch (Exception)` 后静默处理、返回模糊成功结果，或把内部堆栈 / SQL / 配置直接返回给用户时。
- **为什么重要**：前者导致静默失败，后者导致信息泄露。
- **建议如何审**：检查控制器、任务、队列 worker、统一异常处理器、调试输出开关。
- **修复建议**：保留日志上下文并返回受控错误信息，生产环境禁用详细内部错误暴露。
- **何时给代码示例**：建议给。

### ORM / 集合便利写法掩盖真实高成本查询
- **何时报告**：循环中访问关联属性触发 N+1、无分页拉全量数据、集合链式处理掩盖数据库成本时。
- **为什么重要**：在 PHP Web 场景里很容易变成真实性能瓶颈。
- **建议如何审**：关注 Eloquent 关联访问、列表页、后台统计、导出任务、模板中访问关联字段。
- **修复建议**：显式预加载、列裁剪、分页或批处理，不要把 ORM 便利语法误当成零成本。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### 缓存和会话状态一致性不足
- **何时报告**：更新后未失效缓存、会话状态和数据库状态不一致、权限变更后缓存仍放行时。
- **为什么重要**：会造成用户看到旧数据或权限判断错误。
- **修复建议**：明确缓存键、失效时机和会话更新策略。

### 类型和空值语义不清晰
- **何时报告**：数组 / 对象 / null 混用、框架 helper 返回语义不清，已影响调用方正确处理时。
- **为什么重要**：会提高运行时错误和分支遗漏概率。
- **修复建议**：收敛边界返回结构，必要时补充类型声明或值对象。

### 外部调用缺少超时或重试边界
- **何时报告**：HTTP、队列、存储、第三方 SDK 调用没有超时或无限重试，可能放大故障时。
- **为什么重要**：会拖垮请求或 worker。
- **修复建议**：增加超时、重试上限和失败降级策略。

### 过度依赖框架魔法导致行为不透明
- **何时报告**：模型事件、全局作用域、隐式转换、魔术属性等已让行为边界不清、容易误判时。
- **为什么重要**：会提高维护和排障成本。
- **修复建议**：在关键路径上显式表达副作用和依赖关系。

### 调用约定、返回结构或分层边界变更未同步
- **何时报告**：controller、service、action、policy、resource、DTO 或表单验证层的参数顺序、默认值、返回结构、异常语义、字段映射发生变化后，调用点、模板、转换层、中间件或上下游模块没有同步更新时；或绕过既有 service / policy / Form Request 边界直接跨层调用时。
- **为什么重要**：PHP 动态性和框架魔法会让这类问题更容易以隐蔽回归、字段丢失、权限漏校验或错误处理失真形式出现。
- **修复建议**：同步更新调用点与转换层，收紧跨层依赖，必要时补充契约测试、响应结构断言和权限边界测试。

## 5. 低优先级问题（谨慎报告）

### 命名、目录和样板结构可优化
- **何时报告**：仅当控制器过重、服务边界不清、命名混乱，已经影响修改安全性时。
- **为什么重要**：属于维护性问题，不应压过安全和正确性问题。

### 风格一致性与注释问题
- **何时报告**：仅在违反项目规范或当前实现明显难懂时。
- **为什么重要**：不要把 PSR / 注释类问题机械上报成主要审查意见。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非已造成明确问题：

- 纯粹因为代码“不够面向对象”或“不够函数式”就要求重写
- 机械要求所有地方都补满类型声明
- 机械要求把所有数组都替换成 DTO / Value Object
- 没有实际风险时泛泛批评框架魔术方法
- 可由代码风格工具自动处理的低价值问题
- 与当前改动无关的历史目录结构问题

## 7. 该语言特有的审查重点

### Web 安全边界
- PHP 审核里，输入校验、模板输出、文件处理、会话 / Cookie、授权边界是最常见的高价值区域
- Blade / Twig 自动转义并不意味着所有输出都安全，原样输出和富文本尤其要小心

### 框架便利性陷阱
- ORM、helper、魔术属性、中间件和模型事件容易让风险藏在“写起来很方便”的代码里
- 审核时要看真实数据流和权限流，而不是只看语法是否优雅

### 查询与任务执行成本
- N+1、全量加载、导出任务内逐条处理，是 PHP 项目里很常见的实际性能问题

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- SQL 注入、XSS、文件上传、反序列化、越权访问等安全问题
- 异常处理不当、内部错误泄露
- ORM 查询导致的隐蔽高成本路径
- 输入校验边界错误
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 目录和分层整理
- 一般性类型声明增强
- 轻度命名和结构优化
- 非关键路径性能建议

## 9. 审核输出建议

- 优先报告输入边界、安全、授权、异常流和高成本查询问题
- 不要把 PHP reference 写成 PSR 规范或框架教程摘要
- 如果提出性能问题，应说明它会在哪个页面、任务或查询路径上产生实际影响
- 如果提出结构问题，应说明它如何增加回归或维护风险
- 如果没有发现值得报告的高置信度问题，应明确说明

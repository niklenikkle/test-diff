# Java 审核标准

## 1. 适用范围

本参考适用于 Java 代码审核，重点覆盖：

- Spring / Jakarta 服务端代码、后台任务、批处理、CLI 与基础库
- 控制器、服务层、持久层、DTO / VO 转换、事务、异常处理与资源管理
- 并发执行、缓存、数据库访问、外部服务调用与配置边界
- 与 SQL 注入、越权、事务错误、资源泄漏、空指针和性能退化相关的风险点

当变更涉及接口入参、数据库操作、事务边界、线程池、异常处理、配置读取或权限控制时，应优先参考本文件。

## 2. 审核目标

审核 Java 代码时，优先关注：

- 是否存在逻辑错误、空值风险、异常流错误、事务不一致或资源管理问题
- 是否存在输入验证不足、SQL / 命令注入、敏感信息泄露、权限绕过等安全问题
- 是否因为并发、缓存、延迟加载或层次边界处理不当而引入稳定性风险
- 是否出现明显的数据访问低效、阻塞调用或错误恢复策略缺失
- 是否与项目既有异常体系、分层约定和框架使用方式冲突

## 3. 高优先级问题（优先报告）

### 输入未经校验直接进入关键业务或持久层
- **何时报告**：请求参数、DTO、路径参数、消息体或配置值未经校验就进入查询、权限判断、对象创建或状态变更逻辑时。
- **为什么重要**：容易造成脏数据、逻辑绕过或安全漏洞。
- **建议如何审**：关注 controller 边界、`@RequestBody`、`@PathVariable`、分页排序参数、枚举值和 ID 类字段。
- **修复建议**：在入口增加 Bean Validation、白名单限制和语义校验，不要把“类型匹配”当成“输入合法”。
- **何时给代码示例**：建议给。

### SQL / JPQL / 原生查询拼接存在注入风险
- **何时报告**：用户输入通过字符串拼接进入 SQL、JPQL、动态排序字段或过滤条件时。
- **为什么重要**：存在直接注入风险，且在报表、搜索、后台接口中很常见。
- **建议如何审**：查看 `EntityManager`、JDBC、MyBatis 动态 SQL、拼接 order by / like 条件的代码。
- **修复建议**：优先使用参数绑定；字段名、排序方向等无法参数化的部分改用白名单映射。
- **何时给代码示例**：安全问题，建议给。

### 事务边界错误导致数据不一致
- **何时报告**：多步写操作缺少事务、事务传播级别不当、异常被吞掉导致事务未回滚，或在事务中混入不安全的外部副作用时。
- **为什么重要**：会直接造成部分提交、脏状态或补偿困难。
- **建议如何审**：关注 `@Transactional` 边界、自调用失效、批处理、消息发送、库存 / 余额 / 状态变更逻辑。
- **修复建议**：收敛事务边界，明确回滚条件，把外部副作用与数据库事务解耦或补充一致性策略。
- **何时给代码示例**：建议给。

### 异常捕获过宽或错误被吞掉
- **何时报告**：使用 `catch (Exception)` 后仅记录日志或返回默认值，导致真实故障被隐藏或流程继续执行危险路径时。
- **为什么重要**：会造成静默失败、错误状态扩散和排障困难。
- **建议如何审**：检查 service、批任务、重试逻辑、远程调用包装层、统一异常处理器。
- **修复建议**：收窄异常类型，保留上下文，并明确错误是否应中断当前流程。
- **何时给代码示例**：建议给。

### 资源管理或外部调用关闭不完整
- **何时报告**：文件流、数据库资源、HTTP 响应体、线程池、定时任务等资源未及时关闭，或错误路径遗漏清理时。
- **为什么重要**：会导致连接泄漏、线程耗尽和系统不稳定。
- **建议如何审**：关注 try-with-resources、客户端响应关闭、线程池创建位置、定时任务生命周期。
- **修复建议**：优先使用自动关闭机制，明确资源所有权和销毁时机。
- **何时给代码示例**：建议给。

### 权限控制只做了表层检查
- **何时报告**：仅校验“是否登录”或仅在控制器层做粗粒度判断，未对具体资源做授权校验时。
- **为什么重要**：会导致对象级越权访问。
- **建议如何审**：查看订单、文档、组织、后台管理资源是否按当前用户 / 角色 / 租户做显式校验。
- **修复建议**：在资源读取与修改入口增加对象级授权，不要只依赖页面隐藏或前置过滤器。
- **何时给代码示例**：建议给。

### 并发访问共享状态缺少线程安全保证
- **何时报告**：单例 bean、缓存、静态变量、集合、计数器在多线程下被读写且缺少同步时。
- **为什么重要**：会引发竞态条件、状态错乱和间歇性故障。
- **建议如何审**：关注 `@Service` 单例内部字段、异步任务共享对象、缓存刷新、惰性初始化。
- **修复建议**：减少共享可变状态，必要时使用并发集合、锁或原子类。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### N+1 查询、懒加载误用或分页策略不当
- **何时报告**：接口或批任务中存在明显 N+1、无分页拉全表、序列化触发懒加载连锁查询时。
- **为什么重要**：会明显拖慢接口并放大数据库压力。
- **修复建议**：使用 fetch join、批量查询、投影、分页或预加载策略。

### 空值语义不清晰
- **何时报告**：返回 `null`、Optional、异常三种风格混用，导致调用方难以正确处理时。
- **为什么重要**：会增加 NPE 和逻辑分支错误。
- **修复建议**：收敛边界语义，明确“缺失”“失败”“空结果”的区别。

### 远程调用缺少超时、重试或熔断约束
- **何时报告**：HTTP / RPC / MQ 调用没有超时、无上限重试或错误策略不清时。
- **为什么重要**：会造成线程阻塞、级联故障和雪崩。
- **修复建议**：增加超时、重试边界和降级策略，并区分可重试错误。

### 缓存一致性风险
- **何时报告**：写库后未更新缓存、缓存键设计不稳定、批量失效策略可能污染读路径时。
- **为什么重要**：会造成难以察觉的数据陈旧问题。
- **修复建议**：明确读写一致性策略，并验证失效与回填顺序。

### 接口契约、DTO 语义或分层调用变更未同步
- **何时报告**：controller / service / repository / DTO / VO 的参数顺序、默认值、返回结构、异常语义、字段映射或状态语义发生变化后，调用点、装配层、测试或上下游模块没有同步更新时；或跨层直连破坏既有分层约定时。
- **为什么重要**：这类问题经常表现为隐蔽回归、序列化不一致、事务语义偏差或运行时异常，排查成本高。
- **修复建议**：同步更新调用点与装配层，收紧跨层依赖，必要时补充契约测试、序列化断言或边界测试。

## 5. 低优先级问题（谨慎报告）

### 命名、分层与样板代码整理机会
- **何时报告**：仅当命名混乱、层次职责穿透、样板代码已显著妨碍理解和修改时。
- **为什么重要**：属于维护性问题，不应压过事务、安全和正确性风险。

### 风格和注解使用习惯问题
- **何时报告**：仅在违反项目明确规范，或当前用法已导致误解时。
- **为什么重要**：不要把通用 Java 风格建议当主要审查意见。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非已造成明确问题：

- 纯粹因为代码不够“OO”或“不够函数式”就要求重写
- 机械要求所有地方都用 Optional
- 机械要求所有查询都改成 stream 或相反
- 没有实际影响时泛泛批评 getter / setter / DTO 样板代码
- 可由格式化、静态分析自动发现的低价值风格问题
- 与当前改动无关的历史分层设计问题

## 7. 该语言特有的审查重点

### 服务端边界与事务
- Java 审核里高价值问题常集中在 controller → service → repository 链路
- 事务边界、异常传播和对象级授权是否一致，决定很多真实生产风险

### 框架与运行时特性
- Spring 单例 bean 默认共享状态，需要警惕线程安全
- JPA / ORM 的懒加载、事务上下文和序列化交互容易引入隐藏问题

### 资源与并发
- 线程池、连接池、外部调用超时、缓存一致性，是稳定性问题高发区

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- 输入校验缺失、SQL 注入、越权访问
- 事务边界错误、异常吞掉、资源未关闭
- 单例共享状态并发风险
- N+1 或懒加载引发的隐蔽性能 / 正确性问题
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 分层整理
- 命名和样板代码优化
- 一般性 Optional / stream 使用建议
- 非关键路径性能建议

## 9. 审核输出建议

- 优先报告输入边界、安全、事务一致性、异常流和并发问题
- 不要把 Java reference 写成框架最佳实践汇编
- 如果提出性能问题，应说明具体查询路径、数据规模或触发条件
- 如果提出结构问题，应说明它如何增加回归或维护风险
- 如果没有发现值得报告的高置信度问题，应明确说明

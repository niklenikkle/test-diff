# C/C++ 审核标准

## 1. 适用范围

本参考适用于 C / C++ 代码审核，重点覆盖：

- 系统程序、性能敏感模块、底层库、驱动接口、嵌入式与原生服务代码
- 内存管理、对象生命周期、指针 / 引用、资源释放、并发与 ABI 边界
- 字符串处理、数组 / 缓冲区访问、文件 / 网络 I/O、异常与错误码路径
- 与越界访问、悬垂引用、双重释放、未定义行为、竞态条件和注入风险相关的问题

当变更涉及裸指针、手动内存管理、资源句柄、并发逻辑、序列化边界、格式化输出或系统调用时，应优先参考本文件。

## 2. 审核目标

审核 C / C++ 代码时，优先关注：

- 是否存在内存安全问题、未定义行为、资源泄漏或对象生命周期错误
- 是否存在缓冲区溢出、格式化字符串、命令注入、路径风险等明确安全问题
- 是否因为并发、异常路径、所有权不清导致稳定性或数据一致性风险
- 是否出现明显的无界拷贝、重复分配、错误的容器使用等性能问题
- 是否与项目既有所有权约定、错误处理风格和资源管理模型冲突

## 3. 高优先级问题（优先报告）

### 越界访问、缓冲区溢出或长度校验缺失
- **何时报告**：数组、缓冲区、字符串拷贝、序列化读写、协议解析中缺少边界检查，或长度来源不可信时。
- **为什么重要**：会直接造成内存破坏、崩溃甚至可利用漏洞。
- **建议如何审**：关注 `memcpy` / `strcpy` / `snprintf` / 下标访问 / 自定义 parser / 长度字段计算。
- **修复建议**：补齐边界校验，优先使用带长度语义的安全接口和容器，不要信任外部长度值。
- **何时给代码示例**：安全问题，建议给。

### 对象生命周期错误导致悬垂指针、悬垂引用或 use-after-free
- **何时报告**：返回局部对象地址、保存临时对象引用、容器扩容后继续使用失效指针 / 迭代器，或资源释放后继续访问时。
- **为什么重要**：这是 C / C++ 中最常见也最危险的稳定性问题之一。
- **建议如何审**：检查返回值、缓存引用、回调注册、异步任务持有对象、容器重分配点、手动 delete 路径。
- **修复建议**：明确所有权边界，避免暴露短生命周期引用，必要时改用值语义、智能指针或稳定存储。
- **何时给代码示例**：建议给。

### 内存或资源释放协议错误
- **何时报告**：存在内存泄漏、双重释放、释放方式不匹配、异常路径遗漏清理、句柄未关闭时。
- **为什么重要**：会造成长期稳定性问题，严重时触发未定义行为。
- **建议如何审**：关注 `new` / `delete`、`malloc` / `free`、文件句柄、socket、锁、事务对象、自定义析构逻辑。
- **修复建议**：优先使用 RAII 和智能指针，收敛资源所有权，不要依赖人工记忆收尾。
- **何时给代码示例**：建议给。

### 格式化字符串、命令拼接或路径使用不安全
- **何时报告**：用户输入直接作为格式化模板、shell 命令参数、文件路径或动态库路径使用时。
- **为什么重要**：存在格式化字符串漏洞、命令注入、路径遍历或加载错误模块的风险。
- **建议如何审**：查看 `printf` 家族、`system` / `popen`、日志、诊断命令、导入导出路径处理。
- **修复建议**：固定格式串、参数分离、白名单校验和路径归一化，不要把原始用户输入直接送入系统接口。
- **何时给代码示例**：安全问题，建议给。

### 并发共享状态缺少同步或可见性保证
- **何时报告**：多个线程访问共享对象、缓存、标志位或容器，缺少锁、原子操作或内存序保证时。
- **为什么重要**：会引发竞态条件、数据损坏和难以复现的生产故障。
- **建议如何审**：关注后台线程、回调、双缓冲、状态机、惰性初始化、跨线程对象传递。
- **修复建议**：明确并发模型，使用锁、原子类型或线程隔离，不要默认“简单读写没问题”。
- **何时给代码示例**：建议给。

### 错误路径或异常路径遗漏关键清理与状态回滚
- **何时报告**：函数在中途失败时未释放部分资源、未恢复状态、未回滚写入或对象处于半初始化状态时。
- **为什么重要**：会造成泄漏、脏状态或后续链路继续在错误前提下运行。
- **建议如何审**：检查多阶段初始化、早返回、异常抛出、部分成功流程、构造函数失败路径。
- **修复建议**：使用 RAII 封装阶段性资源，把清理绑定到对象生命周期，而不是散落在分支里。
- **何时给代码示例**：建议给。

## 4. 中优先级问题（按上下文报告）

### 不必要的拷贝、重复分配或容器误用
- **何时报告**：热点路径中存在可见的大对象拷贝、频繁 realloc、字符串反复拼接或错误容器选择时。
- **为什么重要**：会带来性能损耗，但应基于实际路径报告。
- **修复建议**：按需使用 move、reserve、string_view、合适容器或对象复用策略。

### 接口所有权语义不清晰
- **何时报告**：函数参数 / 返回值无法判断所有权转移、调用方是否负责释放、引用是否长期有效，已影响安全修改时。
- **为什么重要**：会放大后续 use-after-free 或泄漏风险。
- **修复建议**：通过命名、类型和注释明确所有权，优先使用表达语义更强的类型。

### 错误处理模型混乱
- **何时报告**：错误码、异常、日志返回值混用，导致调用方无法可靠判断失败状态时。
- **为什么重要**：会增加隐式失败和错误传播断裂的风险。
- **修复建议**：统一边界上的错误表达方式，并保留必要上下文。

### 现代 C++ 特性使用不当导致复杂度上升
- **何时报告**：模板、元编程、宏、运算符重载等用法已经明显降低可读性或引入误用风险时。
- **为什么重要**：会增加维护成本，但不应压过真实内存安全问题。
- **修复建议**：收敛抽象层级，优先选择更直接、可验证的实现。

## 5. 低优先级问题（谨慎报告）

### 命名、注释和组织方式可优化
- **何时报告**：仅当接口意图不清、状态机难懂、资源语义不明确，已影响后续修改安全性时。
- **为什么重要**：属于维护性建议，不应压过内存和并发风险。

### 风格和语法偏好问题
- **何时报告**：仅在违反项目明确规范或影响理解时。
- **为什么重要**：不要把花括号风格、`auto` 偏好、头文件顺序之类的问题当主要审查意见。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非已造成明确问题：

- 纯粹因为代码不够“现代 C++”就要求大改
- 机械要求把所有裸指针都替换成智能指针，而不区分所有权语义
- 机械要求所有容器都做极限性能优化
- 没有证据时泛泛批评模板、宏或内联函数“可能有性能问题”
- 可由 clang-format / clang-tidy 直接处理的低价值风格问题
- 与当前改动无关的历史架构问题

## 7. 该语言特有的审查重点

### 内存与生命周期
- 所有权是否清楚、对象是否仍然有效，是 C / C++ 审核中的最高价值问题之一
- 很多 bug 不是“写法不好”，而是会直接落到未定义行为

### 安全边界
- 缓冲区、格式化字符串、路径和系统调用接口需要默认带着怀疑去看
- 外部输入一旦参与长度、地址、命令或格式串拼接，风险通常很高

### RAII 与并发
- 是否真正利用 RAII 封装清理逻辑，决定异常和早返回路径是否可靠
- 线程共享状态的可见性和同步策略要比“有没有加锁”更重要

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- 越界访问、格式化字符串、命令注入等安全问题
- 悬垂指针 / 引用、双重释放、资源泄漏
- 并发竞态和同步错误
- RAII 重构能明显降低错误概率的场景
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 一般性的命名和组织调整
- 非关键路径上的微优化建议
- 抽象层级或现代 C++ 用法收敛建议
- 可读性增强类意见

## 9. 审核输出建议

- 优先报告内存安全、生命周期、并发和系统边界问题
- 不要把 C / C++ reference 写成语法和库用法教程
- 如果提出性能问题，应说明它处于什么热点路径或资源约束下
- 如果提出接口设计问题，应说明它如何放大所有权或生命周期风险
- 如果没有发现值得报告的高置信度问题，应明确说明

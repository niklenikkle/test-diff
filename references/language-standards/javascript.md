# JavaScript / TypeScript 审核标准

## 1. 适用范围

本参考适用于 JavaScript / TypeScript 代码审核，重点覆盖：

- 浏览器端逻辑、React / Next.js 组件与 hooks
- Node.js 服务端代码、API、异步流程
- 类型系统边界、输入输出边界、状态管理、数据获取
- 与 XSS、注入、认证授权、信息泄露相关的风险点

当变更涉及组件状态、hooks、异步调用、类型声明、接口入参、HTML 渲染、服务端边界或数据库访问时，应优先参考本文件。

## 2. 审核目标

审核 JavaScript / TypeScript 代码时，优先关注：

- 是否存在逻辑错误、状态错误、异步错误传播缺失
- 是否存在安全问题，如 XSS、注入、敏感信息泄露、权限绕过
- 是否出现类型逃逸，导致静态检查失效
- 是否引入明显的渲染、请求或数据访问性能问题
- 是否与项目既有模式冲突，造成后续维护成本上升

## 3. 高优先级问题（优先报告）

### 类型逃逸或错误的类型断言
- **何时报告**：使用 `any`、过度断言（如 `as any`、`as unknown as T`）、绕过空值和联合类型检查，且这会削弱关键路径的正确性保障时。
- **为什么重要**：会让 TypeScript 失去约束能力，增加运行时错误概率。
- **建议如何审**：关注接口入参、API 响应、表单数据、状态对象、工具函数返回值是否被粗暴断言。
- **修复建议**：补齐准确类型，优先缩小输入类型范围，必要时增加运行时校验。
- **何时给代码示例**：涉及复杂类型推导、联合类型收窄失败、关键业务对象结构时建议给。

### React hooks 依赖错误或闭包过时
- **何时报告**：`useEffect` / `useMemo` / `useCallback` 依赖缺失、依赖错误，或事件处理器捕获过期状态导致逻辑异常时。
- **为什么重要**：容易引发重复请求、状态不同步、数据陈旧、隐藏性 bug。
- **建议如何审**：关注数据获取、订阅解绑、计时器、回调函数、依赖数组是否与实际使用值一致。
- **修复建议**：补齐依赖，重构副作用边界，必要时拆分 effect 或引入稳定引用。
- **何时给代码示例**：通常建议给，尤其是副作用逻辑复杂时。

### 渲染阶段错误地修改状态
- **何时报告**：在渲染过程中直接调用 `setState`、修改外部可变对象，或在计算路径中引发副作用时。
- **为什么重要**：可能导致无限循环、难以定位的渲染异常或状态污染。
- **建议如何审**：看组件函数体、渲染分支、memo 计算、衍生数据逻辑中是否混入写操作。
- **修复建议**：将状态更新移动到事件、effect 或明确的状态转换流程中。
- **何时给代码示例**：建议给。

### 未经净化地渲染用户输入或 HTML
- **何时报告**：使用 `dangerouslySetInnerHTML`、模板拼接 HTML、将不可信内容直接插入页面时。
- **为什么重要**：存在明显 XSS 风险。
- **建议如何审**：关注富文本渲染、评论区、CMS 内容、Markdown 渲染、URL 参数注入点。
- **修复建议**：改为安全文本渲染，或使用经过审计的净化库如 DOMPurify。
- **何时给代码示例**：安全问题，建议给。

### 服务端输入未经验证直接进入数据库或关键逻辑
- **何时报告**：请求参数、body、query、headers、cookies 未校验即用于查询、权限判断、对象创建或命令拼接时。
- **为什么重要**：容易引发 SQL 注入、逻辑绕过、脏数据落库。
- **建议如何审**：检查 API 边界、控制器、路由处理、DTO/schema、ORM 查询拼接。
- **修复建议**：引入 schema 校验（如 zod / valibot / class-validator），并坚持参数化查询。
- **何时给代码示例**：建议给。

### 访问控制缺失或对象级授权缺失
- **何时报告**：用户可通过 ID、slug、路径参数访问不属于自己的对象，或仅做“已登录”检查但未做资源级授权时。
- **为什么重要**：属于高价值安全问题，常见且影响大。
- **建议如何审**：查看用户资料、订单、文件、组织资源、后台接口是否同时检查身份与资源权限。
- **修复建议**：在对象读取与修改入口加入显式授权判断，不要只依赖前端隐藏按钮。
- **何时给代码示例**：建议给。

### 敏感信息硬编码或错误暴露
- **何时报告**：API key、token、密码、私有地址、内部堆栈、调试信息直接出现在源码或响应中时。
- **为什么重要**：会带来直接泄露风险。
- **建议如何审**：关注配置文件、客户端 bundle、错误处理中间件、日志输出。
- **修复建议**：改用环境变量、密钥管理服务，生产环境隐藏内部错误细节。
- **何时给代码示例**：建议给。

### 外部请求目标未校验
- **何时报告**：服务端根据用户提供 URL 发起请求，且没有域名白名单、协议限制或内网访问防护时。
- **为什么重要**：存在 SSRF 风险。
- **建议如何审**：查看 webhook、抓取器、代理接口、预览接口、文件拉取接口。
- **修复建议**：做域名白名单、协议限制、IP/网段限制，并限制重定向。
- **何时给代码示例**：安全问题，建议给。

## 4. 中优先级问题（按上下文报告）

### 异步错误处理不完整
- **何时报告**：Promise 拒绝未处理、`catch` 为空、错误被吞掉、后台任务失败无监控时。
- **为什么重要**：会造成静默失败和排障困难。
- **修复建议**：补充错误传播、记录和上层处理策略。

### 客户端 / 服务端边界混乱
- **何时报告**：在 Server Component 中使用客户端能力，或把仅服务端可用的数据 / 逻辑泄露到客户端时。
- **为什么重要**：会导致运行时错误、构建失败或敏感信息暴露。
- **修复建议**：明确边界，必要时拆分 client/server 模块。

### 模块边界或调用约定变更未同步
- **何时报告**：函数、hook、组件、API client 或共享类型的参数顺序、默认值、返回结构、异常语义发生变化后，调用点、适配层或上下游模块没有同步更新时；或跨层直接依赖破坏既有模块边界时。
- **为什么重要**：这类问题常表现为隐蔽回归、运行时错误或类型系统形同虚设，且排查成本高。
- **修复建议**：同步更新调用点与适配层，收紧共享契约，必要时补充边界测试或兼容层。

### 列表 key 不稳定
- **何时报告**：对可重排、可插入、可删除的列表使用索引作为 key，且可能导致状态错位时。
- **为什么重要**：会引发 UI 状态异常。
- **修复建议**：使用稳定唯一标识作为 key。

### 明显的重复渲染或昂贵计算未隔离
- **何时报告**：热点组件中存在可见的重复计算、重复创建对象/函数、或昂贵子组件反复渲染时。
- **为什么重要**：会引发可感知的性能退化。
- **修复建议**：按需使用 `useMemo`、`useCallback`、`React.memo`，或重构组件边界。

### 数据访问低效
- **何时报告**：存在明显 N+1 查询、无分页、无列裁剪、无必要 limit，或请求链重复拉取相同数据时。
- **为什么重要**：会拖慢接口并放大系统成本。
- **修复建议**：改为批量查询、JOIN、缓存或分页策略。

## 5. 低优先级问题（谨慎报告）

### 命名、注释、JSDoc 不理想
- **何时报告**：仅当其明显影响理解，或项目规范明确要求时。
- **为什么重要**：属于维护性问题，不应喧宾夺主。

### 轻度结构优化机会
- **何时报告**：当函数过长、组件职责过多、重复代码已开始影响后续修改时。
- **为什么重要**：这类问题值得提，但通常不应压过安全和正确性问题。

### 风格一致性问题
- **何时报告**：仅在违反项目既有规则或工具配置时。
- **为什么重要**：不要把 prettier / eslint 可自动处理的问题当作主要审查意见。

## 6. 不要轻易报告的内容

以下内容默认不应成为主要审查意见，除非它们已经造成明确问题：

- 纯个人风格偏好，如分号、引号、链式写法偏好
- 不影响行为的“更优雅写法”建议
- 单纯因为存在内联箭头函数就上报性能问题
- 对小型组件机械要求 `useCallback` / `useMemo`
- 对当前改动无关的历史代码不一致性
- 可由 lint / formatter 自动修复的低价值问题

## 7. 该语言特有的审查重点

### JavaScript
- 运行时类型缺失，输入边界容易失守
- 异步流程复杂，错误传播容易遗漏
- 原型链、动态对象结构、隐式转换可能带来隐藏 bug

### TypeScript
- 重点审查类型是否真的在保护关键路径，而不是只是“看起来有类型”
- 警惕 `any`、过度断言、宽泛泛型、错误的类型收窄

### React / Next.js
- hooks 依赖、状态闭包、渲染副作用、服务端 / 客户端边界是高价值问题集中区
- 用户输入渲染、路由参数、数据获取状态、缓存边界值得重点关注

### Node.js
- API 边界输入验证、错误处理、权限控制、超时、资源释放、第三方服务调用是关键审核点

## 8. 何时提供代码修改示例

以下情况建议提供“修改前 / 修改后”的代码示例：

- 安全漏洞，如 XSS、注入、权限绕过、SSRF
- hooks 依赖错误、渲染期状态修改等隐蔽逻辑 bug
- 复杂类型问题，单靠文字不易说清
- API 边界输入验证与错误处理缺失
- 用户明确要求直接给出修复方案

以下情况通常只给修改方向即可：

- 轻度命名优化
- 一般性的结构整理
- 未明显造成问题的小型性能建议

## 9. 审核输出建议

- 优先报告少量高价值问题，不要把本文件当作逐条打卡清单
- 安全和正确性问题优先于风格与轻度重构建议
- 如果没有发现值得报告的高置信度问题，应明确说明
- 如果提出性能问题，应尽量说明触发条件或实际影响，而不是泛泛而谈
- 如果提出类型问题，应明确指出它会导致什么运行时风险

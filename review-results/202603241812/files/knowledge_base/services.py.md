## 问题详情

**问题 1：动态 SQL 直接拼接表名、字段和值，可被构造为任意 SQL 注入**
- **文件位置**: `knowledge_base/services.py:388-400`
- **问题描述**: `execute_dynamic_query` 使用 f-string 拼接 `table_name` 和 `conditions`，再直接执行 `cursor.execute(sql)`。
- **影响分析**: 调用方只要控制 `conditions` 中的 key 或 value，即可闭合字符串并拼接额外 SQL 条件。即使表名做过格式限制，字段名和值仍完全未参数化。该接口直接跨过 ORM 和参数绑定，是当前项目最危险的执行面之一。
- **优先级**: 高
- **修复建议**: 禁止直接拼接 SQL；表名与字段名使用显式白名单，值必须通过参数化查询绑定，并限制该能力仅对受信任内部调用开放。

**问题 2：查询异常直接把堆栈返回给调用方，泄露内部实现细节**
- **文件位置**: `knowledge_base/services.py:226-235`
- **问题描述**: 发生异常时，返回消息中包含 `str(e)` 和完整 `traceback.format_exc()`。
- **影响分析**: 调用方可直接看到文件路径、调用链、数据库错误文本甚至业务数据片段，这会显著降低攻击者摸清内部结构的成本。
- **优先级**: 高
- **修复建议**: 对外仅返回通用错误消息，把详细堆栈保留在受控日志中。

**问题 3：按路径导入 JSON 文件，允许服务端读取任意可访问文件**
- **文件位置**: `knowledge_base/services.py:360-386`
- **问题描述**: `import_from_file` 直接对外部传入路径执行 `open(file_path)` 并 `json.load`。
- **影响分析**: 只要上层接口可匿名或低权限访问，攻击者就能借此探测服务器文件系统，尝试读取任意 JSON 文件，或借由报错侧信道获知文件存在与格式信息。
- **优先级**: 高
- **修复建议**: 只接受上传文件对象或受控目录中的文件标识，不允许客户端直接指定服务器绝对路径。

**问题 4：查询日志记录完整返回结果，包含 SQL 与潜在敏感数据**
- **文件位置**: `knowledge_base/services.py:439-441`
- **问题描述**: `log_query_details` 将用户问题和完整结果 JSON 直接写入日志。
- **影响分析**: 返回结果中包含 `sql_query`、匹配问题、相似问题等业务敏感信息，若日志被收集、导出或共享，会把知识库内容整体暴露给日志读取方。
- **优先级**: 高
- **修复建议**: 日志只保留必要的追踪字段，如请求 ID、匹配状态、耗时和知识 ID，避免记录完整 SQL 与响应正文。

**问题 5：相似度匹配对全部知识逐条做 Python 级遍历，数据增长后会明显退化**
- **文件位置**: `knowledge_base/services.py:118-141`
- **问题描述**: `find_best_match` 先取出全部启用知识，再逐条归一化、分词和计算相似度。
- **影响分析**: 当前实现是典型的全表扫描 + CPU 密集比较，知识量增长后查询延迟会线性上升，且每次请求都会重复计算历史数据的归一化文本和关键词。
- **优先级**: 中
- **修复建议**: 先用关键词、标签或预计算索引缩小候选集，再做相似度精排，并尽量缓存问题归一化结果。

**问题 6：分类统计存在 N+1 查询模式**
- **文件位置**: `knowledge_base/services.py:327-339`
- **问题描述**: 先遍历 `Category.objects.all()`，再对每个分类分别执行两次 `count()`。
- **影响分析**: 分类数量增加时，数据库请求数会随之线性增长，统计接口会越来越慢，且对数据库造成不必要压力。
- **优先级**: 中
- **修复建议**: 使用聚合或注解一次性返回各分类总数和启用数，避免循环内重复查询。

**问题 7：全量导出与动态查询结果均无上限，容易造成大响应和内存压力**
- **文件位置**: `knowledge_base/services.py:341-358`, `knowledge_base/services.py:398-411`
- **问题描述**: `export_all_knowledge` 直接遍历全部知识，`execute_dynamic_query` 直接 `fetchall()` 全部结果。
- **影响分析**: 当知识库或目标表数据量较大时，请求会占用大量内存并拖慢进程，严重时会影响服务稳定性。
- **优先级**: 中
- **修复建议**: 为导出和动态查询增加分页、条数上限或流式输出机制。

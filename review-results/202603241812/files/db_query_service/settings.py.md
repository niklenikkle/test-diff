## 问题详情

**问题 1：源码中直接硬编码多组真实凭据与 API Key**
- **文件位置**: `db_query_service/settings.py:9`, `db_query_service/settings.py:66-69`, `db_query_service/settings.py:146-156`, `db_query_service/settings.py:178-185`
- **问题描述**: Django `SECRET_KEY`、MySQL 账号密码、SMTP 密码、API_KEYS 以及额外数据库连接密码均直接写在源码中。
- **影响分析**: 一旦代码被共享、备份或上传到仓库，这些凭据会直接泄露，攻击者可伪造会话、连接数据库、调用外部接口或横向进入其他系统。该问题已实际落在信任边界上，不是理论风险。
- **优先级**: 高
- **修复建议**: 立即将所有密钥和密码迁移到环境变量或专用密钥管理系统，并视为已暴露凭据执行轮换。

**问题 2：服务默认关闭认证和权限，导致所有 API 默认可匿名访问**
- **文件位置**: `db_query_service/settings.py:102-104`
- **问题描述**: `DEFAULT_AUTHENTICATION_CLASSES` 和 `DEFAULT_PERMISSION_CLASSES` 被配置为空列表。
- **影响分析**: 在 DRF 中这等价于所有未单独声明权限的接口都允许匿名访问。结合项目里存在写操作、批量更新、历史清空、导出和调试接口，意味着外部请求无需登录即可修改或读取系统数据。
- **优先级**: 高
- **修复建议**: 至少为全局接口启用认证和默认权限，并仅对确需公开的只读接口做显式放开。

**问题 3：允许任意 Host 与任意来源跨域，放大未授权调用面**
- **文件位置**: `db_query_service/settings.py:13`, `db_query_service/settings.py:87-88`
- **问题描述**: `ALLOWED_HOSTS = ['*']` 且 `CORS_ALLOW_ALL_ORIGINS = True`，同时还允许携带凭据。
- **影响分析**: 任意域名都可以在浏览器中直接调用该服务；若后续接入 Cookie、Session 或管理端登录，跨站调用面会立刻扩大。结合当前全局匿名权限，任何来源都可直接发起管理型请求。
- **优先级**: 高
- **修复建议**: 仅保留受信任的 Host 和前端域名白名单，并避免将“允许所有来源”和“允许凭据”同时用于生产配置。

**问题 4：生产安全开关整体关闭，导致敏感流量可被明文和弱头保护处理**
- **文件位置**: `db_query_service/settings.py:163-170`
- **问题描述**: `SESSION_COOKIE_SECURE`、`CSRF_COOKIE_SECURE`、`SECURE_SSL_REDIRECT`、HSTS 和 `SECURE_CONTENT_TYPE_NOSNIFF` 均关闭。
- **影响分析**: 一旦服务被部署到真实环境，Cookie 可经明文链路传输，浏览器安全头缺失，HTTPS 不被强制，整体安全基线明显偏低。
- **优先级**: 高
- **修复建议**: 按环境拆分配置，生产环境开启 HTTPS 重定向、安全 Cookie、HSTS 与基础安全头。

**问题 5：日志级别固定为 DEBUG 且写文件，容易把敏感请求内容长期落盘**
- **文件位置**: `db_query_service/settings.py:108-144`
- **问题描述**: `django` 与 `knowledge_base` logger 默认都写入 `debug.log`，级别为 `DEBUG`。
- **影响分析**: 结合业务层会记录查询结果、环境变量和用户行为，调试级日志会显著增加敏感信息留存范围，后续排障文件也更难控管。
- **优先级**: 中
- **修复建议**: 区分开发/生产日志级别，并为包含敏感字段的业务日志增加脱敏与最小化记录策略。
